2. 讨论问题、问题时需要站在相关问题专家的角度上进行分析，避免过度迎合且用户能力不足造成了整体方案质量不够，降低效率
3. 始终使用中文进行回复

# 文档要求

## API接口文档

1. 文档位置：docs\API接口文档.md
2. 处理要求：所有涉及增删改API的都需要维护API接口文档的对应内容，方便后续查看

## 测试文档

1. 测试文档统一参考：`docs\测试文档\测试用例书写规则.md`
2. `AGENTS.md` 只保留测试流程和引用关系，不重复展开完整测试用例细则
3. 新增或维护测试文档时，按功能存放到 `docs\测试文档\[功能名称]\` 目录下

## db文档

1. 数据库说明文档：记录数据库之间的表之间，表和功能的关系。存放在 `docs\db\` 目录下
2. 数据库表文档：每个表有自己的文档，记录表结构和字段说明。存放在 `docs\db\[表名称]\` 目录下
3. 新增或修改数据库时文档时，需要维护对应的数据文档

# Git 操作

## 操作规范要求

1. **生成注释**：在暂存后，必须自动生成一段简洁、语义化的中文提交信息（Commit Message），采用 `feat:` 或 `fix:` 等约定式格式，仅供用户参考。

## 依次上传操作规范

### 指令触发：

当用户要求“提交改动”或“依次提交改动”时
当接收到此指令时，请严格按以下逻辑执行操作：

### 指令步骤

1. 意图识别与范围锁定

- 识别改动：首先运行 git status 和 git diff --name-only。
- 会话隔离：仅处理本次会话中明确涉及修改的文件。如果工作区存在其他未提及的改动，必须逐一询问我是否需要包含，严禁自行处理。

2. 分功能原子化操作 (Atomic Staging)
   不要一次性 add 所有文件，请按以下步骤循环执行：

- 逻辑归类：将修改的文件按功能逻辑分组（例如：UI 调整为一组，API 逻辑为一组）。
- 精确暂存：针对每一组功能，使用 git add `<file1>` `<file2>` ...。
- 生成 Commit 建议：
  按照 类型(模块): 简短描述 格式生成。
  例如：feat(auth): 增加微信登录回调接口 或 fix(styles): 修复移动端顶部遮挡。

3. 提交和交互

- 依次执行提交并推送
- 状态反馈：每提交完一组功能，输出如下格式：
  📦 已暂存功能组 [N]
  文件：path/to/file
  建议注释：feat: ...

4. 收尾检查：

- 所有属于本次会话的文件暂存完成后，输出：
  ✅ 本次会话改动已全部分批推送
- 若有问题请输出具体问题

### 安全规范

1. 严禁越界：绝对禁止对非本次会话的文件执行 add、commit 或 checkout。
2. 禁止全堆叠：禁止将所有文件打包在一个 Commit 中，必须体现分功能的原子性，方便后续回滚。

# 自动化全栈闭环开发规则

### 1. 需求设计 (写方案)

方案定稿后，AI 必须主动且只在此刻将完整的技术方案写入 `docs/需求文档/[功能名称].md`，并
**要求：需求文档必须强制包含以下四块结构，严禁分离测试文档：**

- **模块一：【需求概述与交互】**：需求背景 + 一句话需求描述 + 解决方案大纲 + 用户的核心体验链路。（核心是让所有没有上下文的人都知道遇到什么问题？要干什么？）
- **模块二：【前端与组件设计】**：新增或修改的路由、需要编写/复用的 React 组件。
- **模块三：【后端与架构设计】**：新增或修改的 API 设计、相关数据库的 Schema 结构增删改查设计、新增或修改的埋点设计。
- **模块四：【📝 联合自动化验收用例 (TDD核心)】**：采用 `- [ ]` 待办列表格式，明文规定后续 AI 必须**逐一编写代码自测**的清册。

  - **第一阶段用例（Node 接口与数据库闭环）**：比如“拦截无效脏数据，断言 400，并查库确认脏标志未更改”。
  - 每个涉及数据库的功能，至少要包含 1 条“接口级验证”用例和 1 条“数据库字段级验证”用例。
    - “数据库字段级验证”用例必须写明：表名、字段名、预期值、关联关系，不允许只写“查库确认成功”。
  - **第二阶段用例（E2E 端到端浏览器闭环）**：比如“无脑输入手机号 -> 倒计时 -> 提交 -> 拦截跳转”。
- **页面结构与线框方案（按需强制产出）**：

  - 凡是涉及新页面、页面结构重构、复杂交互、信息架构调整、工作台/看板/后台/编辑器类界面的需求，在方案阶段除需求文档外，必须额外产出一份《页面结构与线框方案》。
  - 该文档的作用等同于传统 PRD 中的低保真原型/结构原型，核心目标是让后续阅读者和执行 AI 明确知道页面长什么样、区域如何划分、主要交互入口和链路如何组织。
  - 若仅为纯接口改动、后端逻辑修复、测试补齐或小范围样式微调，则不强制单独产出页面结构与线框方案。

### 2. 方案确认

1. 用户提出修改意见，AI修改方案的对应内容
2. 确认方案，上述开发点和测试点同步更新 `todo.md`。
3. 更新docs目录下的数据库文档、埋点文档

### 3. AI 代码开发 (写代码)

AI 进入 `EXECUTION` 模式，依据第一阶段生成的《需求文档》分别在前后端进行正常的底层逻辑代码编写。

---

### 4. AI 技术自测 (开发自测)

开发完成后，AI 先进行技术层自测，再进入正式测试用例编写与执行，禁止跳过技术自测直接交付用户。

- **动作 1：后端闭环验证**
  AI 必须执行 Node 脚本完成接口与数据库黑箱自测。必须包含：

  - **A. 接口级验证**：模拟客户端发起 HTTP 真实请求（涵盖正/反例），验证 JSON 结构与状态码。
  - **B. 数据库验证**：请求结束后，脚本须用 SQL/ORM 强行直连查库并输出至终端，自证数据零脏点 100% 精确落位。
  - **C. 字段级核验要求**：凡涉及数据库写入的功能，查库结果必须至少覆盖主表主键、关联键、关键业务字段；页面通过或接口 200 不能替代数据库闭环。
- **动作 2：前端闭环验证**
  组装页面后，AI 必须基于 Playwright 等 E2E 无头模拟器跑通预设的前端操作链路，作为最后一道防线。

  - 若功能涉及明显的 UI 布局、视觉层级、动效、空态/结果态切换、报告页展示等主观界面体验，AI 还必须额外生成本地截图产物用于人工 UI 回归。
  - UI 截图回归的**强制触发条件**仅限以下场景：页面布局层级调整、视觉样式重构、空态/加载态/结果态变化、报告页/详情页/弹窗/表单主交互变化。纯接口、纯后端、纯数据库或不影响结构的微小文案改动，不强制要求截图。
  - UI 截图回归的**最小截图集**默认是 3 张：输入态/默认态、加载态/处理中、结果态/成功态；只有存在明显异常分支时，才额外补 1 张异常态。
  - 截图文件命名统一使用可读状态名，例如：`xxx-input.png`、`xxx-loading.png`、`xxx-result.png`、`xxx-error.png`，禁止使用无语义编号命名。
  - 截图产物统一放到 `docs/测试文档/[功能名称]/UI/` 目录，至少覆盖关键状态（如输入态、加载态、结果态/异常态），文件名需可直接看懂用途。
  - 该截图产物默认用于本地验收留档，除非用户明确要求或项目已有归档约定，否则不应默认纳入 commit。
- **动作 3：查缺补漏与自治循环**
  若任一脚本报错、报红，AI 必须内部消化：**自行查日志 -> 自行修代码 -> 自动重跑测试**，直至核心技术自测通过。

---

### 5. 正式测试用例编写与执行

完成技术自测后，AI 必须补齐正式测试用例文档，并按照测试用例再执行一轮完整自测与修复。

- **动作 1：编写正式测试用例**

  - 测试用例书写规范统一参考：`docs/测试文档/测试用例书写规则.md`
  - 测试文档按功能落到：`docs/测试文档/[功能名称]/`
  - 用例需覆盖页面、API、数据库、异常与人工核心链路，不允许只写页面层描述
  - 若功能涉及数据库，测试文档必须显式包含以下独立模块：
    - 页面主链路测试
    - API 接口测试
    - 数据库存储与写入测试
    - 异常与边界测试
  - “数据库存储与写入测试”中至少要校验：写入是否发生、字段是否正确、关联是否正确、是否存在脏数据或遗漏字段
- **动作 2：按用例执行自测**

  - AI 必须依据正式测试用例逐项执行验证
  - 对已执行项回填结果；未执行项不得伪造通过状态
  - 若存在失败项，必须在对应模块底部记录原因
  - 若数据库模块未执行或未完成字段级查库，不得把整项功能标记为“已完整验收”
  - 若本轮包含 UI 截图回归，需在测试文档或执行记录中写明截图目录、覆盖状态与是否用于人工复核
  - 若本轮不满足 UI 截图回归触发条件，应在执行记录中明确写明“本轮未触发 UI 截图回归”，避免遗漏与误判
- **动作 3：问题修复与回归**

  - 对测试用例执行中发现的问题，AI 必须先修复，再按受影响用例回归
  - 未完成“按用例自测并修复”的功能，不应进入人工验收阶段

---

### 6. AI 闭环测试结果输出 (机器验收)

技术自测与正式测试用例执行完成后，AI 才能输出最终自测结果。

- **动作 1：输出自测报告（交付前置）**
  全部自动化动作执行完毕后，AI 必须输出一份结构化“自测报告”并附在交付消息中，至少包含：
  - **A. 功能范围与变更文件**：本次覆盖的功能点、核心改动文件列表。
  - **B. 执行环境**：本地/CI、Node 版本、数据库环境、关键环境变量。
  - **C. 测试执行清单**：`lint`、`typecheck`、集成测试、E2E 的执行命令与结果（通过/失败）。
  - **C1. 正式测试用例执行情况**：测试文档路径、已执行范围、通过/失败/阻塞概况。
  - **C2. UI 回归截图产物（如有）**：截图目录、覆盖状态、是否已供人工验收查看。
  - **D. 失败项明细（如有）**：失败用例、报错摘要、日志/trace 路径、根因判断、是否已修复、剩余风险。
  - **E. 数据库验证结果**：必须列出本次核验的表名、主键/关联键、关键字段、实际查询结果摘要、是否存在脏数据。
  - **F. 最终结论**：`✅ Verified` 或 `❌ Blocked`；若 Blocked 必须写明阻塞点与下一步建议。
  - 若功能涉及数据库但未完成直连查库，不得输出 `✅ Verified`。

---

### 7. 交付人工验收 (人工体验)

只有当技术自测、正式测试用例编写、按用例自测修复、自测报告输出全部完成后，AI 方可通知用户进入人工验收。用户仅对界面像素设计、色块美观和主观操作感受进行修改体验。
完成体验后，AI 在用户授权下分组暂存 `git commit`。
