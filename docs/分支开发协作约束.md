# 分支开发协作约束

## 1. 文档目的
多人同时开发时，冲突通常不是 Git 本身造成的，而是功能边界没有提前切开。本文档用于约束前端、后端 API、数据库和文档的开发边界，避免不同人员在同一批核心文件里互相覆盖。

适用场景：
- 新功能由不同人员并行开发。
- 一个功能涉及前端页面、后端接口、数据库表、脚本和测试文档。
- 需要把“结果筛选、商品数据整理、Ozon 上货”等连续链路拆开开发。

## 2. 分支约束
### 2.1 分支命名
建议使用以下格式：

- `feature/[功能英文名]`
- `fix/[问题英文名]`
- `docs/[文档英文名]`
- `test/[测试英文名]`

示例：
- `feature/product-data-prep-api`
- `feature/product-data-prep-ui`
- `feature/product-publish-draft-db`
- `docs/product-data-prep-contract`

### 2.2 分支职责
一个分支只负责一个清晰目标。

允许：
- 一个分支只做商品数据整理 API。
- 一个分支只做商品数据整理前端页面。
- 一个分支只做草稿表和数据库文档。

不建议：
- 一个分支同时改结果页、商品整理页、Ozon 上货页、后端 API 和数据库。
- 一个分支同时做 UI 重构、接口语义变更和数据迁移。

### 2.3 合并前要求
合并前必须确认：
- 已同步目标分支最新代码。
- 已执行本功能相关构建或自测。
- 没有提交非本功能文件。
- 文档、API 文档、数据库文档、测试文档与代码一致。

## 3. 前端开发约束
### 3.1 页面边界
新功能优先新增独立页面或独立组件，不直接改已有复杂页面。

商品数据整理工作台约束：
- 页面入口：`/product-data-prep`
- 页面文件：`frontend/menglar-workbench/src/pages/ProductDataPrepPage.jsx`
- 不在 `ResultsPage.jsx` 中堆商品整理逻辑。
- 不在 `OzonUploadPage.jsx` 中堆商品整理逻辑。

### 3.2 组件拆分
进入正式开发后，商品数据整理建议拆为：
- `ProductPrepBatchSelector`
- `ProductPrepTable`
- `ProductPrepEditor`
- `ProductImageWorkbench`
- `ProductPrepValidationPanel`
- `ProductDraftExportPanel`

每个开发者只认领明确组件，避免多人同时修改一个大组件。

### 3.3 路由与侧边栏
`main.jsx` 和 `AppShell.jsx` 属于高冲突入口文件。

约束：
- 新路由和侧边栏入口由一个人集中修改。
- 其他人不要反复调整导航顺序、文案和路由路径。
- 若必须改入口文件，先同步最新代码并在提交说明里写清原因。

### 3.4 样式约束
样式改动优先使用功能前缀，避免污染全局。

商品数据整理样式建议统一前缀：
- `.product-prep-*`

禁止：
- 为新功能随意修改 `.wb-panel`、`.wb-table`、`.wb-page` 等全局样式。
- 在同一个提交里混入其他页面的样式调整。

## 4. 后端 API 开发约束
### 4.1 命名空间隔离
商品数据整理后端统一使用独立命名空间：

- `/api/product-data-prep`

建议接口：
- `GET /api/product-data-prep/candidates`
- `POST /api/product-data-prep/drafts`
- `GET /api/product-data-prep/drafts`
- `GET /api/product-data-prep/drafts/:id`
- `PATCH /api/product-data-prep/drafts/:id`
- `POST /api/product-data-prep/drafts/:id/validate`
- `POST /api/product-data-prep/drafts/export`

禁止：
- 修改 `/api/products` 的既有筛选语义来承载商品整理逻辑。
- 修改 Ozon 上货执行接口来承载商品整理逻辑。
- 把草稿保存、图片整理、属性映射都塞进一个旧接口。

### 4.2 后端文件拆分
如果当前服务入口集中在 `backend/menglar-workbench-api/server.mjs`，新功能不应继续在入口文件中堆业务代码。

建议结构：
- `backend/menglar-workbench-api/routes/product-data-prep.mjs`
- `backend/menglar-workbench-api/services/product-data-prep-service.mjs`
- `backend/menglar-workbench-api/repositories/product-data-prep-repository.mjs`

`server.mjs` 只负责注册路由，尽量只产生少量挂载代码。

### 4.3 API 契约先行
接口开发前必须先明确请求和响应结构。

草稿对象建议基础结构：

```json
{
  "id": 1,
  "sourceJobId": 11,
  "productNormalizedId": 123,
  "ozonProductId": "210798735",
  "title": "",
  "description": "",
  "categoryId": null,
  "attributes": [],
  "images": [],
  "price": null,
  "stock": null,
  "draftStatus": "draft",
  "validationIssues": []
}
```

接口字段稳定前，前端先使用 mock 数据联调；字段稳定后再接真实数据库写入。

### 4.4 API 文档要求
所有新增、删除、修改 API 都必须同步维护：

- `docs/API接口文档.md`

API 文档至少包含：
- 接口路径
- 请求方法
- 请求参数
- 请求体结构
- 响应结构
- 错误码或错误返回
- 是否读库或写库

## 5. 数据库开发约束
### 5.1 原始数据不可覆盖
商品数据整理不能直接覆盖采集和筛选结果。

禁止写回：
- `products_raw`
- `products_normalized`

允许读取：
- `source_jobs`
- `products_normalized`

商品整理后的结果必须写入独立草稿表。

### 5.2 草稿表建议
建议新增：
- `product_publish_drafts`
- `product_publish_draft_images`
- `product_publish_draft_attributes`
- `product_publish_draft_validation_issues`

核心关联：
- `product_publish_drafts.source_job_id -> source_jobs.id`
- `product_publish_drafts.product_normalized_id -> products_normalized.id`
- `product_publish_draft_images.draft_id -> product_publish_drafts.id`
- `product_publish_draft_attributes.draft_id -> product_publish_drafts.id`
- `product_publish_draft_validation_issues.draft_id -> product_publish_drafts.id`

### 5.3 字段级验证要求
涉及数据库写入时，自测必须查库验证字段级结果。

至少验证：
- 主表主键
- 来源任务 ID
- 来源商品 ID
- 草稿状态
- 标题
- 描述
- 类目 ID
- 价格
- 库存
- 图片 URL
- 属性 ID 和属性值
- 校验问题记录

不能只用接口 200 或页面显示成功代替查库。

### 5.4 数据库文档要求
新增或修改数据库时必须维护：

- 数据库说明文档：`docs/db/数据库说明文档.md`
- 表文档：`docs/db/[表名称]表文档.md` 或 `docs/db/[表名称]/`

文档至少包含：
- 表用途
- 字段说明
- 主键
- 外键或逻辑关联
- 状态枚举
- 读写接口
- 测试验证重点

## 6. 前后端联调约束
### 6.1 推荐顺序
1. 先写需求文档和接口契约。
2. 前端基于 mock 数据开发页面。
3. 后端基于契约开发接口。
4. 数据库表稳定后接入真实读写。
5. 联调后补接口测试、数据库测试和 E2E 测试。

### 6.2 并行开发边界
前端人员：
- 不修改后端业务语义。
- 不临时要求后端复用旧接口写入新数据。
- 不把接口字段写死到多个组件里。

后端人员：
- 不修改前端页面结构。
- 不改变已有接口响应字段含义。
- 不直接写回原始商品表。

数据库人员：
- 不在未通知前端和后端的情况下改字段名。
- 不删除已有字段。
- 新字段必须提供默认值或兼容策略。

## 7. 测试与验收约束
### 7.1 技术自测
涉及后端和数据库时，必须执行：
- HTTP 接口真实请求测试。
- 数据库直连查询验证。
- 字段级核验。

涉及前端页面时，必须执行：
- 构建测试。
- 浏览器路由测试。
- 关键交互 E2E 测试。

涉及 UI 结构变化时，必须生成截图，放到：

- `docs/测试文档/[功能名称]/UI/`

### 7.2 测试文档
测试文档统一放到：

- `docs/测试文档/[功能名称]/`

测试用例应覆盖：
- 页面主链路测试
- API 接口测试
- 数据库存储与写入测试
- 异常与边界测试
- UI 截图回归说明

## 8. 提交约束
### 8.1 原子提交
按功能分组提交，不要一次性打包所有文件。

建议提交顺序：
- `docs(product-data-prep): 补充商品整理接口契约`
- `feat(db): 新增商品发布草稿表`
- `feat(api): 新增商品数据整理草稿接口`
- `feat(workbench): 接入商品数据整理页面`
- `test(api): 补充商品数据整理接口自测`

### 8.2 禁止越界提交
提交前必须执行：

```bash
git status --short
git diff --name-only
git diff --cached --name-only
```

确认只暂存本功能相关文件。

禁止：
- `git add .`
- 把其他人的未提交改动一起提交。
- 在修一个功能时顺手格式化大文件。
- 在同一提交里混入无关样式、文档和脚本。

### 8.3 提交信息
提交信息使用中文约定式格式：

- `feat(模块): 简短描述`
- `fix(模块): 简短描述`
- `docs(模块): 简短描述`
- `test(模块): 简短描述`
- `refactor(模块): 简短描述`

## 9. 商品数据整理专项红线
商品数据整理功能开发必须遵守以下红线：

1. 不能修改 `/api/products` 的既有筛选语义。
2. 不能写回 `products_normalized`。
3. 不能把整理逻辑塞进 Ozon 上货执行接口。
4. 不能大改 `server.mjs`，只能做路由挂载。
5. 不能把候选商品、草稿、校验、导出混成一个接口。
6. 不能在前端把商品整理逻辑塞进 `ResultsPage.jsx` 或 `OzonUploadPage.jsx`。
7. 不能在没有数据库字段级验证的情况下宣称完整验收。

## 10. 最小安全开发方案
如果多人并行开发商品数据整理，推荐拆成四条线：

### 10.1 前端线
负责：
- `/product-data-prep` 页面。
- mock 数据展示。
- 草稿列表、编辑器、校验面板。

不负责：
- 后端接口实现。
- 数据库表结构。
- Ozon 真实提交。

### 10.2 后端 API 线
负责：
- `/api/product-data-prep/*` 接口。
- 参数校验。
- 草稿读写服务。
- 导出结构生成。

不负责：
- 前端页面布局。
- 直接调用 Ozon 真实上货。
- 改写原始商品表。

### 10.3 数据库线
负责：
- 草稿表设计。
- 数据库文档。
- 迁移脚本。
- 字段级验证脚本。

不负责：
- 页面交互。
- API 路由命名。

### 10.4 测试线
负责：
- 接口测试脚本。
- 数据库查库验证。
- E2E 测试。
- UI 截图回归。
- 测试文档回填。

不负责：
- 修改业务实现，除非明确修复测试发现的问题。
