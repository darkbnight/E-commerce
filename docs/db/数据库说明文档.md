# 数据库说明文档

## 1. 目标

本数据库用于承接萌拉数据网站第一阶段的商品基础数据抓取结果，服务于以下能力：

- 浏览器自动化抓取任务记录
- 页面原始数据存档
- 商品标准化数据沉淀
- 后续筛选、分析和导出

## 2. 当前表设计

当前阶段只保留 3 张表：

### `source_jobs`

记录每次抓取任务的来源页面、执行状态和结果摘要。

当前阶段把“来源页面配置”直接并入这张表，不单独拆 `source_pages`。

采集结果摘要分两层：

- 商品级兼容字段：`raw_count`、`normalized_count`、`warning_count`，用于热销商品入库链路。
- 跨任务统一字段：`request_count`、`success_count`、`record_count`、`error_type`，用于行业采集、商品采集等不同任务类型在任务页统一展示。

### `products_raw`

记录抓取任务过程中保存的原始数据载荷，用于回溯字段变化、排查解析失败和后续重新清洗。

### `products_normalized`

记录标准化后的商品基础数据，用于后续筛选、分析和导出。

## 3. 表关系

- `source_jobs` 1 -> N `products_raw`
- `source_jobs` 1 -> N `products_normalized`
- `products_raw` 1 -> 0..1 `products_normalized`

说明：

- 一条原始记录可能成功标准化为一条商品数据。
- 也可能因为字段缺失、结构变化或解析失败，暂时无法产出标准化记录。

## 4. 设计取舍

### 为什么当前不单独设计 `source_pages`

如果后续需要长期维护多个固定抓取入口，例如“热销榜”“潜力榜”“关键词榜”“店铺榜”，单独拆 `source_pages` 会更适合，因为它可以复用页面名称、URL、分页方式、登录要求等配置。

但当前阶段目标只是验证：

- 紫鸟浏览器登录态是否可被自动化复用
- 是否能抓到基础商品数据
- 是否能稳定完成原始入库和标准化入库

在这个阶段把页面信息直接记录到 `source_jobs` 更直接，能减少一层抽象和一张表的维护成本。等后续真的出现多个稳定抓取入口，再拆 `source_pages` 更合理。

## 5. 功能对应

- 抓取任务管理：`source_jobs`
- 原始数据追溯：`products_raw`
- 商品基础数据查询：`products_normalized`

## 6. 后续扩展方向

第二阶段如果开始建设筛选能力，数据库会在现有 3 张表基础上继续扩展，例如：

- 筛选规则表
- 筛选结果表
- 标签表
- 页面来源配置表

当前阶段不提前落这些表，避免在抓取链路尚未跑通前过度设计。
