# 数据库说明文档

## 1. 数据库定位

电商工作台数据库用于承接采集任务、商品经营快照、商品内容资产、商品 SKU 资产和原始采集审计数据。数据库不绑定单一数据源，后续可以接入萌拉、Ozon、1688、广告数据或其它电商运营来源。

目标数据库名称：

```text
db/ecommerce-workbench.sqlite
```

历史数据库名称：

```text
db/menglar-mvp.sqlite
```

旧名称仅作为迁移来源保留，不再作为后续默认数据库名称。

## 2. 核心数据分层

当前数据库按业务对象拆成六类核心数据：

| 数据层 | 表名 | 作用 |
| --- | --- | --- |
| 采集任务数据 | `source_jobs` | 记录一次采集行为的来源、执行状态和结果摘要，不承担来源专属流程状态机 |
| 商品经营数据 | `product_business_snapshots` | 记录某个平台商品在某一次采集任务中的经营快照，如销量、销售额、曝光、点击、物流和尺寸重量 |
| 商品筛选工作流 | `product_selection_items` | 记录商品进入筛选工作台后的阶段状态、测价结果、供应链和流转状态 |
| 商品内容数据 | `product_content_assets` | 记录商品标题、描述、标签、主图、多图等产品级内容资产，并保留内容版本能力 |
| 商品 SKU 数据 | `product_content_skus` | 记录某一条内容资产版本下面的 SKU 集合，包括平台 SKU ID、SKU 名称、价格和图片数组 |
| 原始审计数据 | `products_raw` | 保存原始采集载荷，用于排错、回放和重新解析 |

## 3. 表关系

```text
source_jobs
  1 -> N products_raw

source_jobs
  1 -> N product_business_snapshots

source_jobs
  1 -> N product_content_assets

source_jobs
  1 -> N product_content_skus

products_raw
  1 -> 0..1 product_business_snapshots

product_business_snapshots
  通过 platform + platform_product_id
  关联同一商品的多次经营快照

product_selection_items
  N -> 1 product_business_snapshots
  通过 source_snapshot_id 关联来源经营快照

product_selection_items
  N -> 1 source_jobs
  通过 source_job_id 关联来源采集任务

product_content_assets
  通过 platform + platform_product_id
  关联同一商品的多次内容版本

product_content_skus
  N -> 1 product_content_assets
  通过 content_asset_id 关联所属内容版本
```

## 4. 商品身份规则

当前阶段不单独创建 `product_master` 商品主表。商品身份由稳定来源字段表达：

```text
platform + platform_product_id
```

含义如下：

- `platform`：平台标识，如 `ozon`
- `platform_product_id`：该平台内的商品 ID

如果某个来源暂时没有稳定的平台商品 ID，才允许退化使用：

```text
platform + normalized_product_url
```

不建议使用品牌、标题、规格等字段做商品主去重，因为这些字段容易变化，也容易误把相似商品识别成同一商品。

## 5. 内容版本规则

内容资产采用“商品身份 + 内容指纹”的建模方式：

- 同一个商品允许存在多条内容版本
- 是否为同一版内容，主要靠 `content_hash` 判断
- 是否为最新版本，主要靠 `captured_at` 判断
- 当前阶段不强制落库显式 `version_no`

推荐规则：

1. 先按 `platform + platform_product_id` 找到商品身份
2. 再对标题、描述、标签、主图、多图和 SKU 资产计算 `content_hash`
3. 如果新采集内容与该商品最新版本 `content_hash` 相同，则不新增内容版本
4. 如果 `content_hash` 不同，则新增一条内容资产，并写入该版本对应的一组 SKU

## 6. 迁移原则

本次结构仍属于开发期演进，但原则上保持来源无关：

1. 数据库默认文件名从 `db/menglar-mvp.sqlite` 调整为 `db/ecommerce-workbench.sqlite`
2. `products_normalized` 停止作为目标业务表使用
3. 新建 `product_business_snapshots`，承接经营快照数据
4. 调整 `product_content_assets`，只承接产品级内容资产
5. 新建 `product_content_skus`，承接 SKU 级资产和内容版本映射关系
6. `source_jobs` 保持轻量任务摘要，不新增来源专属状态机字段
7. `products_raw` 继续作为原始审计表保留

## 7. 页面与功能对应

| 功能 | 主要读取表 | 说明 |
| --- | --- | --- |
| 采集任务页 | `source_jobs` | 展示一次采集行为的状态、数量、错误和执行时间 |
| 结果展示页 | `source_jobs` + `product_business_snapshots` | 按批次展示经营快照，用于浏览原始采集结果和基础筛选 |
| 商品筛选页 | `product_selection_items` + `product_business_snapshots` | 以经营快照为主视角推进初筛、测价、供应链和流转 |
| 内容资产查询 | `product_content_assets` + `product_content_skus` | 读取某商品最新内容版本及其 SKU 集合，用于后续内容整理和上货准备 |
| 数据排错 | `products_raw` | 对照原始载荷和解析结果，定位字段变化或解析失败 |

## 8. 后续扩展

如果后续进入上货准备和长期商品管理阶段，再新增以下表，不建议提前塞进当前核心内容表：

| 表名 | 作用 |
| --- | --- |
| `product_publish_drafts` | 上货草稿、标题、类目、价格、库存、物流等发布字段 |
| `product_image_downloads` | 更细粒度的图片下载任务和文件状态 |
| `source_pages` | 固定采集入口配置，适用于稳定维护多个采集入口后再拆 |
