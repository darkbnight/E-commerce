# 云端数据库与多人协作方案

## 1. 文档目标

本文用于统一多人开发、云端数据库、真实业务数据和数据库结构变更的协作规则。

当前项目需要同时解决两个问题：

- 开发过程中数据库结构会持续变化，需要可追踪、可测试、可发布。
- 真实业务数据需要多人共享，用于采集、筛选、分析和经营决策。

结论是：不要用 Git 同步 SQLite 数据文件，而是使用 Git 管代码和 migrations，使用云端 PostgreSQL 管真实业务数据。

## 2. 总体方案

采用以下分层：

```text
Git 仓库：
代码、数据库迁移脚本、种子数据、脱敏样例、数据库文档

本地开发数据库：
开发验证、结构试错、自动化测试、临时采集缓存

云端 PostgreSQL：
多人共享真实业务数据、准生产或生产数据、长期数据资产
```

核心原则：

- `.sqlite` 数据文件不进入 Git。
- 数据库结构变化必须通过 migrations 记录。
- 云端数据库不重建真实数据，只通过增量 migration 升级结构。
- 真实业务数据保存在云端 PostgreSQL，不通过 Git 同步。
- 本地和云端使用同一套代码、同一套 migrations，通过不同 `DATABASE_URL` 切换数据库。

## 3. 分支职责

### 3.1 main 分支

`main` 是业务协作分支，用于查看和操作真实线上或准线上业务数据。

职责：

- 保持相对稳定。
- 对应云端真实业务数据库。
- 用于业务决策、真实数据查看、稳定采集和业务验收。
- 只接收已经在 `dev` 分支验证通过的改动。

限制：

- 不直接承接未验证功能。
- 不直接试验数据库结构。
- 不允许把本地临时调试代码直接合入。
- 不允许绕过 migration 直接依赖新表或新字段。

### 3.2 dev 分支

`dev` 是开发集成分支，负责承接多人协作、功能集成、冲突处理和发布前验证。

职责：

- 所有功能分支先合并到 `dev`。
- 在 `dev` 上解决代码冲突、接口冲突、数据库 migration 顺序冲突。
- 在 `dev` 环境完成任务运行验证。
- 确认功能、迁移、测试文档都完整后，再发布到 `main`。

限制：

- `dev` 可以连接本地开发数据库或专用 dev 数据库。
- `dev` 不应直接对云端真实业务数据库做高风险试验。
- `dev` 中未验证通过的 migration 不允许发布到 `main`。

### 3.3 功能分支

功能分支用于单一功能开发。

推荐命名：

```text
feature/[功能英文名]
fix/[问题英文名]
docs/[文档英文名]
test/[测试英文名]
```

合并路径：

```text
feature/*
  -> dev
  -> main
```

禁止路径：

```text
feature/*
  -> main
```

除非是紧急修复，并且明确说明原因、影响范围和验证结果。

## 4. 分支与数据库环境对应关系

推荐对应关系：

```text
feature/*：
本地开发数据库，允许重建和试错

dev：
本地 PostgreSQL 或共享 dev PostgreSQL，用于多人集成验证

main：
云端 shared/prod PostgreSQL，用于真实业务数据查看和稳定运行
```

更清晰的环境命名：

```text
local/dev：
个人开发环境，可重建、可清空、可插入测试数据

shared/dev：
多人集成环境，用于验证合并后的代码和 migrations

shared/prod：
云端真实业务数据库，用于真实业务数据和决策
```

如果当前阶段成本有限，可以先只保留：

```text
本地 dev 数据库
云端 shared/prod 数据库
```

但发布到 `main` 前，仍然必须完成本地或 dev 数据库的 migration 验证。

## 5. 标准协作流程

### 5.1 功能开发流程

```text
1. 从 dev 拉出功能分支
2. 在功能分支开发代码
3. 如涉及数据库结构，新增 migration
4. 在本地开发数据库执行 migration
5. 运行功能自测和必要脚本
6. 合并到 dev
7. 在 dev 分支解决冲突并集成测试
8. dev 验证通过后，发布到 main
9. main 对应云端数据库执行已验证的 migration
10. 使用 main 代码连接云端数据库做业务验证
```

### 5.2 dev 到 main 的发布门禁

从 `dev` 合并到 `main` 前，必须确认：

- 代码已在 `dev` 合并完成。
- migrations 在 dev 数据库执行成功。
- 依赖新表、新字段的功能已完成接口或脚本验证。
- 文档已同步，包括数据库文档、API 文档、测试文档。
- 高风险 migration 已完成备份和演练。
- 已明确云端数据库需要执行哪些 migration。

如果任一项未完成，不应发布到 `main`。

## 6. 云端数据库如何合并和测试

数据库本身不能像代码一样通过 Git 合并真实数据。云端数据库的“合并”指的是：把多个功能分支产生的 migration 在 `dev` 分支统一排序、统一验证，然后再应用到云端数据库。

### 6.1 migration 合并规则

当多个功能分支都修改数据库结构时：

- 各分支都必须提交独立 migration。
- 合并到 `dev` 时检查 migration 文件顺序。
- 如果两个 migration 修改同一张表或同一字段，需要人工确认最终结构。
- 不允许多人分别手工改云端表结构。
- 不允许只改本地数据库而不提交 migration。

推荐 migration 命名：

```text
db/migrations/20260423_001_init_schema.sql
db/migrations/20260423_002_add_product_ai_score.sql
db/migrations/20260424_001_add_source_job_status.sql
```

### 6.2 dev 数据库测试

合并到 `dev` 后，应先在 dev 数据库执行：

```text
npm run db:migrate:dev
```

验证内容：

- migration 是否能从旧结构升级到新结构。
- 新增表是否存在。
- 新增字段类型、默认值、索引是否正确。
- 旧数据是否仍可读取。
- 新功能是否可以写入新字段或新表。
- 失败后是否能定位到具体 migration。

如果 dev 数据库测试失败：

- 不允许发布到 `main`。
- 先修复 migration 或代码。
- 修复后重新执行 dev 验证。

### 6.3 测试数据原则

dev 数据库可以使用：

- 模拟数据。
- 脱敏数据。
- 从云端导出的脱敏样例。
- 本地采集的临时数据。

dev 数据库不应该直接承载唯一真实业务数据。真实业务数据以云端 `shared/prod` 为准。

## 7. 云端数据库如何部署

当前阶段不要求把整个服务部署到云端，可以只部署云端 PostgreSQL。

部署关系：

```text
本地服务 / 当前生产机器 / AI 采集工具
        |
        | DATABASE_URL
        v
云端 PostgreSQL
```

云端数据库部署需要完成：

- 创建 PostgreSQL 实例。
- 创建业务数据库。
- 创建最小权限数据库账号。
- 配置连接白名单或安全访问策略。
- 配置备份策略。
- 在本地 `.env` 或运行环境中配置 `DATABASE_URL`。
- 提供 `.env.example`，但不提交真实连接串。

示例：

```text
DATABASE_URL=postgresql://user:password@host:5432/ecommerce
```

安全要求：

- `.env` 不进入 Git。
- 数据库密码不写入文档。
- 云端数据库账号按用途区分，至少区分管理员账号和应用账号。
- 生产或准生产库执行 migration 前必须确认连接目标。

## 8. 云端数据库如何升级

云端数据库不会自动知道当前代码需要什么结构。它依赖 migration 流程升级结构。

基本机制：

```text
1. Git 仓库保存 db/migrations/
2. 云端数据库保存已执行 migration 记录
3. 执行迁移命令时，对比本地 migrations 和云端记录
4. 只执行云端尚未执行的 migration
5. 执行成功后写入 migration 记录表
```

示例 migration 记录表：

```text
schema_migrations
```

升级流程：

```text
1. 确认 dev 分支已验证通过
2. 合并 dev 到 main
3. 备份云端数据库
4. 使用 main 分支代码连接云端数据库
5. 执行云端 migration
6. 验证 schema_migrations 记录
7. 验证核心表、字段、索引
8. 运行关键业务接口或采集脚本
9. 确认 main 可以读取真实业务数据
```

推荐命令形态：

```text
npm run db:migrate:cloud
```

关键顺序：

```text
先升级云端数据库结构
再使用依赖新结构的 main 代码连接云端数据库
```

如果顺序反了，可能出现：

```text
column does not exist
relation does not exist
接口 500
采集写入失败
页面加载失败
```

## 9. Migration 风险分级

### 9.1 低风险迁移

通常可以在 dev 验证后对云端执行：

- 新增表。
- 新增可空字段。
- 新增默认值安全的字段。
- 新增索引。
- 新增不影响历史数据的配置表。

### 9.2 高风险迁移

必须先备份、评估并尽量拆分步骤：

- 删除表。
- 删除字段。
- 修改字段类型。
- 修改字段含义。
- 批量清洗真实业务数据。
- 拆表、合表。
- 修改主键、唯一约束、外键约束。

高风险迁移不建议直接在云端手工执行，应先在本地或 shared/dev 数据库演练。

## 10. 真实业务数据处理原则

真实业务数据不进入 Git。

允许进入 Git 的内容：

- 表结构 migration。
- 初始化基础数据。
- 枚举和配置模板。
- 脱敏样例数据。
- 可重复执行的数据导入脚本。
- 数据库文档。

不应进入 Git 的内容：

- 客户数据。
- 订单数据。
- 真实商品采集结果。
- 真实采购链接和经营备注。
- 账号、Cookie、Token、连接串。
- 未脱敏的业务报表。

建议对采集数据分层：

```text
products_raw / raw_collected_items：
保存原始采集载荷，允许重复、允许脏数据，用于追溯和重新解析

product_* / business_*：
保存清洗、筛选、审核后的业务数据
```

这样 AI 采集可以高频运行，但不会直接污染正式业务表。

## 11. 关键问题答疑

### 11.1 改用 PostgreSQL 后，是否必须把整个服务部署到云端？

不必须。

只要本地服务、采集脚本或 AI 工具能访问云端 PostgreSQL，就可以继续在本地运行。当前阶段可以只部署云端数据库。

### 11.2 只部署数据库到云端，本地读写会不会受影响？

会有网络延迟和网络稳定性影响，但普通业务读写通常可接受。

需要注意：

- 配置数据库连接白名单或访问权限。
- 不要一条数据建立一次连接。
- 批量采集结果建议批量写入。
- 写入失败时保留本地临时文件、日志或重试机制。

### 11.3 本地开发结构和云端结构冲突，会不会导致数据异常？

会有风险。

风险不来自 PostgreSQL，而来自没有受控的改表流程。解决方式是所有结构变化必须走 migration，云端禁止直接手工乱改表。

### 11.4 功能代码改了，但云端数据库没有执行 migration，会发生什么？

如果新功能依赖新表或新字段，云端没有升级结构时会报错。

常见结果：

- 接口 500。
- SQL 查询失败。
- 采集任务写入失败。
- 页面无法加载数据。
- 数据字段缺失。

所以涉及数据库结构的功能发布顺序必须是：

```text
备份云端数据库
-> 执行 migration
-> 运行或部署新代码
-> 验证核心接口和关键数据写入
```

### 11.5 云端数据库如何保持最新？

不是通过重新构建数据库，也不是通过部署完整 Git 项目。

正确方式是：

```text
拉取最新 main
-> 使用 main 中已验证的 migrations
-> 连接云端数据库
-> 执行未执行过的 migrations
```

云端数据库中的真实数据会保留，结构按 migration 增量升级。

### 11.6 本地 SQLite 是否还需要保留？

可以保留作为过渡或开发调试库，但不应作为多人协作的真实业务数据库。

更长期的建议是：

```text
本地开发也使用 PostgreSQL
云端共享也使用 PostgreSQL
```

这样可以减少 SQLite 和 PostgreSQL 在 SQL 语法、字段类型、索引、时间处理、JSON 支持上的差异。

## 12. 当前阶段建议落地步骤

第一阶段：

- 保留 `.gitignore` 中对 `db/*.sqlite` 的忽略。
- 建立 `dev` 和 `main` 分支职责。
- 新增 PostgreSQL 连接配置。
- 建立 `DATABASE_URL` 环境变量切换机制。
- 建立 `db/migrations/` 目录。
- 明确 migration 执行命令。

第二阶段：

- 将当前 SQLite 表结构整理为初始 PostgreSQL migration。
- 将必要的基础数据整理为 seed。
- 在本地或 shared/dev 数据库执行 migration 验证。
- 将真实业务数据从 SQLite 迁移到云端 PostgreSQL。
- 完成一次 dev 到 main 的数据库升级演练。

第三阶段：

- 采集脚本支持连接云端 PostgreSQL。
- 对采集结果增加 raw 层和清洗层分离。
- 对高风险迁移建立备份和回滚流程。
- 根据业务成熟度决定是否部署云端服务。

## 13. 推荐命令形态

后续可以在项目中补充类似命令：

```text
npm run db:migrate:local
npm run db:migrate:dev
npm run db:migrate:cloud
npm run db:seed:local
npm run db:backup:cloud
```

命令的具体实现需要结合后续选定的迁移工具确定。

可选工具方向：

- Prisma Migrate。
- Drizzle Kit。
- Knex migrations。
- node-pg-migrate。
- 自定义 SQL migration runner。

当前阶段先统一原则：`dev` 负责集成和验证，`main` 负责稳定业务协作；云端数据库结构通过 migration 增量升级，真实业务数据不通过 Git 同步。
