# 商品筛选工作台可读性优化 - 测试用例

## 页面主链路测试

- [ ] 进入 `results?mode=screening` 后，页面展示“商品筛选工作台”。
- [ ] 商品列表使用 `.selection-decision-card` 卡片结构展示，不再依赖 `.selection-table tbody tr`。
- [ ] 每个卡片展示商品图、标题、平台商品 ID、品牌 / 店铺、来源批次、采集类型、采集时间。
- [ ] 每个卡片展示测价 / 利润信息：成本、运费、预估售价、利润判断。
- [ ] 每个卡片展示供应链 / 竞品信息：供应链状态、供应商、流转状态。
- [ ] 每个卡片右侧展示主按钮“查看竞品详情”和当前阶段对应操作按钮。
- [ ] 顶部不再展示入池总数、待测价、待找供应链、待整理竞品、可流转统计卡片。
- [ ] 选品工作流过滤区展示在页面顶部原统计区域，不再作为左侧栏展示。
- [ ] 阶段快捷选择按钮条仍然展示，用户可以直接点击“全部 / 待初筛 / 待测价 / 待找供应链”等阶段。
- [ ] 商品筛选工作台内筛选控件和操作按钮字号可读，不应出现按钮文字过小的问题。
- [ ] 供应链 / 竞品信息区不展示“流转”状态。
- [ ] 点击加入筛选池、阶段推进等成功动作后，不展示绿色成功提示条。
- [ ] 商品卡片内商品标题、指标名称、指标数值和按钮文字明显可读。

## API 接口测试

- [ ] 执行 `node scripts/test-product-selection-api.mjs`，确认商品加入筛选池、重复加入、阶段推进和列表读取接口仍通过。
- [ ] 确认 `GET /api/product-selection/items` 返回字段能满足卡片渲染，不需要新增接口字段。

## 数据库存储与写入测试

- [ ] 表名：`product_selection_items`。
- [ ] 字段：`id`、`source_snapshot_id`、`selection_stage`、`initial_delivery_cost`、`transfer_to_prep_at`。
- [ ] 预期：UI 卡片结构调整不改变写库逻辑；加入筛选池后 `source_snapshot_id` 关联 `product_business_snapshots.id`，自动运费写入 `initial_delivery_cost`。
- [ ] 预期：进入商品数据整理后，`transfer_to_prep_at` 写入非空时间。
- [ ] 预期：点击“测价通过”后，已有 `initial_delivery_cost` 保持不变，不被前端简化公式覆盖。
- [ ] 预期：未发现因 UI 调整产生的重复记录或脏数据。

## E2E 操作测试

- [ ] 执行 `node scripts/test-ecommerce-db-ui.mjs`。
- [ ] 在结果展示中加入一个商品到筛选池。
- [ ] 切换到商品筛选页后，断言 `.selection-decision-card` 只出现 1 条。
- [ ] 点击“查看竞品详情”，断言竞品详情弹窗出现并展示销售表现、流量与转化、广告与物流、尺寸与来源。
- [ ] 关闭弹窗后，继续执行阶段流转并刷新页面，断言最终状态仍保持。

## 异常与边界测试

- [ ] 当筛选池为空时，仍展示既有空态文案和“回到结果展示”按钮。
- [ ] 当过滤条件无匹配商品时，仍展示既有空态文案和“重置过滤”按钮。
- [ ] 当商品缺图片时，卡片展示占位图，不影响操作。
- [ ] 当运费、预估售价或供应商为空时，卡片展示 `-`、`待测价` 或 `未记录`，不阻塞阶段动作。
