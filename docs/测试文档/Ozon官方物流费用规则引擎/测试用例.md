# Ozon 官方物流费用规则引擎测试用例

## 1. 测试范围

- 本地物流规则文件校验：`config/shipping/rules.json`、`config/shipping/fx.json`
- 官方 XLSX 自动解析：`data/shipping-sources/*.xlsx` -> `config/shipping/rules.generated.json`
- 当前生效规则：`config/shipping/rules.json` 已使用官方生成规则，当前服务数为 149
- 官方物流服务校验：只允许 Ozon 官方文档或官方计算器中出现的真实服务
- 后端物流 API：`/api/shipping/methods`、`/api/shipping/rule-info`、`/api/shipping/calculate`、`/api/shipping/calculate-batch`、`/api/shipping/compare`
- 前端页面：`/shipping-calculator`

## 2. 执行环境

- 操作系统：Windows
- Node.js：`v22.14.0`
- 前端构建：Vite 8
- 浏览器自动化：Playwright Chromium
- 数据来源：本地 JSON 规则文件，不依赖数据库

## 3. 页面主链路测试

- [x] 打开 `/shipping-calculator`，页面左侧展示商品尺寸、重量、地理范围、货币价格等输入区。
- [x] 使用默认参数提交，页面右侧展示按价格排序的物流服务列表。
- [x] 服务卡片展示服务名称、官方副标题、价格、时效和官方变体。
- [x] 点击“价格从高到低”后，服务列表按费用降序展示。
- [x] 输入 `2000g` 后提交，页面仍展示可用服务，且包含 `China Post eParcel Economy`。
- [x] 输入 `5000g` 后提交，页面仍展示可用服务，且包含 `China Post eParcel Economy`。
- [x] 输入 `40000g` 后提交，页面展示无可用服务空态，不抛出前端异常。

## 4. API 接口测试

- [x] `GET /api/shipping/methods` 返回当前可用物流方法列表。
- [x] `GET /api/shipping/rule-info` 返回规则文件摘要、方法数量和汇率信息，当前 `methodCount = 149`。
- [x] `POST /api/shipping/calculate` 输入合法参数返回 200，结果包含实重、体积重、计费重和总物流成本。
- [x] 默认样例 `1x1x1cm / 50g / CNY / China -> Shenzhen -> Russia` 中，`China Post to PUDO Economy` 输出 `3.2 CNY`，时效 `13-20 天`。
- [x] 默认样例中，`GUOO Economy Extra Small` 输出 `4.3 CNY`，并包含 `Courier` 与 `PUDO` 两个官方变体。
- [x] 默认样例中，`CEL Economy Extra Small` 输出 `4.42 CNY`，时效 `20-25 天`。
- [x] `POST /api/shipping/calculate` 输入超出指定服务限制的参数返回 400，错误信息包含限制原因。
- [x] `POST /api/shipping/calculate-batch` 同时处理成功项和失败项，返回 `successCount` 与 `failedCount`。
- [x] `POST /api/shipping/compare` 返回可用服务列表，并按 `totalLogisticsCost` 升序排列。
- [x] `POST /api/shipping/compare` 输入 `2000g` 返回可用服务，且包含 `China Post eParcel Economy`。
- [x] `POST /api/shipping/compare` 输入 `5000g` 返回可用服务，且包含 `China Post eParcel Economy`。

## 4.1 规则导入脚本测试

- [x] 执行 `npm run shipping:import` 后生成 `config/shipping/rules.generated.json`。
- [x] 生成文件包含 149 条物流服务规则。
- [x] 生成文件包含 12 个承运商。
- [x] 生成文件包含 CEL 16 条服务规则。
- [x] 生成文件包含 China Post 7 条服务规则。
- [x] 生成文件将官方计算器展示的 `Courier` / `PUDO` 写入 `variants`，不拆成独立承运商。
- [x] 生成文件包含 5000g 档位边界样本：`500g`、`501g`、`2000g`、`2001g`、`5000g`。
- [x] 执行 `npm run shipping:validate:generated` 后生成规则文件结构校验通过。

## 5. 数据库存储与写入测试

本功能第一期不使用数据库存储物流规则，也不写入业务表，因此本轮无数据库写入测试项。

## 6. 异常与边界测试

- [x] 规则不匹配时接口返回 400。
- [x] 汇率按订单日期命中，返回结果包含实际命中的汇率日期。
- [x] 输入超出所有服务限制时，`/api/shipping/compare` 返回空可用列表，不抛出 500。
- [x] 500g 仍可命中 Extra Small 类服务。
- [x] 501g 不再命中 Extra Small 类服务，切入 Small / Budget 等更高重量段服务。
- [x] 2000g 仍可命中可用服务。
- [x] 2001g 切入 Big / Budget 等更高重量段服务。
- [x] 5000g 仍可命中可用服务。
