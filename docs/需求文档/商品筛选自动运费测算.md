# 商品筛选自动运费测算

## 模块一：【需求概述与交互】
- 需求背景：当前商品从结果页进入“商品筛选”工作台后，测价卡片中的“运费”默认为空，运营需要再人工触发测价，导致入池后的第一眼无法判断物流成本。
- 一句话需求描述：所有进入商品筛选池的商品，在创建筛选条目时自动调用本地物流计算器写入初始运费。
- 解决方案大纲：后端在 `POST /api/product-selection/items` 创建条目时，读取快照中的均价、长宽高、重量，调用本地物流比价逻辑，优先取已校准服务中的最低总物流成本写入 `initial_delivery_cost`；若没有已校准服务，则回退到 XLSX 候选服务；若缺少尺寸或重量则保持为空但不阻塞入池。
- 用户的核心体验链路：结果页选择商品 -> 加入筛选池 -> 切到“商品筛选” -> 直接看到该商品的“运费”已自动带出，无需先点“进入测价”。

## 模块二：【前端与组件设计】
- 不新增路由，不新增页面。

## 模块三：【后端与架构设计】
- 修改接口：`POST /api/product-selection/items`
  - 创建筛选条目时补充自动物流测算。
  - 入参不变。
  - 返回结构不变，但 `items[].initialDeliveryCost` 可能在创建后立即有值。
- 物流测算规则：
  - 输入来源：`product_business_snapshots.avg_price_cny`、`length_cm`、`width_cm`、`height_cm`、`weight_g`。
  - 固定参数：`originCountry=CN`、`warehouseType=seller_warehouse`、`salesScheme=realFBS`。
  - 日期优先级：`source_jobs.finished_at` -> `product_business_snapshots.captured_at` -> `product_business_snapshots.created_at` -> 当前日期。
  - 取值策略：优先使用已校准服务中的最低 `totalLogisticsCost`；若为空则再放开 XLSX 候选服务；仍无结果则不写入运费。
- 数据库 Schema：不新增字段，不改表结构，仅在 `product_selection_items.initial_delivery_cost` 上提前写值。
- 埋点设计：本次不新增埋点。
