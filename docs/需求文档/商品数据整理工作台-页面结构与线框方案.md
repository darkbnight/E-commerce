# 商品数据整理工作台-页面结构与线框方案

## 1. 页面定位
该页面位于“商品筛选”和“Ozon 上货工具”之间，负责把候选商品整理成发布草稿。它不是新的真实上货入口，也不应该直接修改采集原始数据。

## 2. 页面区域划分
### 顶部说明区
- 标识：`Product Data Prep`
- 标题：`商品数据整理`
- 说明：承接筛选后的候选商品，整理标题、描述、属性、图片和上货字段。
- 右侧状态卡：说明当前是开发交接页，目标链路是 `筛选商品 -> 发布草稿`。

### 工作流区
- 阶段 1：导入筛选商品
- 阶段 2：整理发布字段
- 阶段 3：发布前校验
- 阶段 4：生成上货草稿

### 安全开发边界区
- 说明前端、后端、数据库、上货工具之间的边界。
- 说明后续多人开发时如何减少冲突。

## 3. 后续完整页面建议
当进入正式开发后，页面可以扩展为三栏结构：

- 左侧：候选批次与整理状态筛选。
- 中间：待整理商品列表。
- 右侧：当前商品编辑器与校验问题。

## 4. 线框草图
```text
+--------------------------------------------------------------------------------+
| Product Data Prep                                                              |
| 商品数据整理                                      [开发交接页] [筛选商品->草稿] |
| 承接筛选后的候选商品，整理标题、描述、属性、图片和上货字段。                  |
+--------------------------------------------------------------------------------+
| 工作流                                                                         |
| +----------------+ +----------------+ +----------------+ +----------------+   |
| | 1 导入筛选商品 | | 2 整理发布字段 | | 3 发布前校验   | | 4 生成草稿     |   |
| +----------------+ +----------------+ +----------------+ +----------------+   |
+--------------------------------------------------------------------------------+
| 安全开发边界                                                                   |
| - 新路由独立开发，不改结果页大表格                                             |
| - 新接口使用 /api/product-data-prep                                             |
| - 新草稿表承接整理结果，不覆盖 products_normalized                             |
| - 先 mock 联调，再接真实数据库写入                                              |
+--------------------------------------------------------------------------------+
```

## 5. 冲突规避建议
- 路由和侧边栏由一人一次性接入，后续不要多人同时编辑 `main.jsx` 和 `AppShell.jsx`。
- 商品整理页拆成多个子组件后，按组件认领开发。
- 后端接口按资源拆文件，不把候选、草稿、校验、导出都堆在 `server.mjs` 一个大段里。
- 数据库迁移和 db 文档由同一人维护，避免表结构和文档不一致。
- 上货工具只读取草稿输出，不反向承担整理逻辑。
