# 萌拉游客权限与登录态稳定化方案

## 模块一：【需求概述与交互】

### 需求背景

当前萌拉采集链路已经具备基础前置检查能力，但在实际使用中，“紫鸟里看起来已登录”和“采集脚本可稳定复用业务接口授权”并不等价。最近多次出现以下问题：

- 页面肉眼可访问，但采集脚本打开目标页后被判定为游客态。
- 能进入页面，但没有捕获到带 Authorization 的萌拉业务接口请求。
- 能捕获请求，但关键业务接口返回 `401/403`。
- 采集任务执行后入库 0 条，且任务状态存在误判风险。

根因不是单一的“未登录”，而是登录态、游客态、接口授权、紫鸟 profile 落盘时机、浏览器复用方式几类问题混在一起。

### 一句话需求描述

将“萌拉登录态是否稳定可采”抽象成独立能力，并为失败场景提供更细粒度诊断、自动恢复和人工修复辅助，避免每次采集前靠临时排查。

### 解决方案大纲

1. **统一判断入口**
   - 所有采集动作都先走“登录态稳定性检查模块”。
   - 只有检查结果为 `ready`，才允许进入正式采集。

2. **细分失败类型**
   - 现阶段的 `guest_blocked` 过于粗糙，需要拆分成更明确的问题类型。
   - 输出应该直接告诉用户是“页面游客态”“跳登录页”“接口没授权”“接口 401/403”还是“profile 未落盘”。

3. **增加自动恢复链路**
   - 失败后先尝试自动刷新 profile 并重检。
   - 必要时切换检查目标页或增加页面交互触发。
   - 记录恢复前后差异，避免重复无效动作。

4. **引入登录修复助手**
   - 当自动恢复失败时，辅助用户快速修复：
     - 自动打开紫鸟到指定萌拉页面
     - 提示用户完成登录/授权
     - 轮询检查是否恢复为可采状态

5. **提升根本稳定性**
   - 长期方案应减少“复制 profile + 系统 Chrome 复用”的依赖。
   - 更稳的方向是直接在紫鸟环境内完成检查和采集，降低授权链丢失概率。

### 用户核心体验链路

1. 用户进入“数据采集”页。
2. 系统先执行登录态稳定性检查。
3. 若结果为 `ready`，允许执行行业数据或热销商品采集。
4. 若结果为 `blocked`，直接展示问题类型和处理建议。
5. 系统先尝试自动恢复；自动恢复失败后，再引导用户执行人工修复。
6. 用户修复后点击重新检查，直到状态变为 `ready`。

## 模块二：【前端与组件设计】

### 页面范围

- `/tasks` 数据采集页

### 组件规划

1. `LoginHealthCard`
   - 展示是否可采、检查时间、实际页面、授权请求数、401/403 数量、处理建议。

2. `LoginRecoveryActions`
   - 提供：
     - `检查是否可采`
     - `刷新 Profile 后检查`
     - `打开紫鸟修复登录`
     - `重新检查`

3. `LoginHealthTimeline`
   - 展示最近一次检查、自动恢复动作、恢复前后差异。

4. `LoginBlockingReasonPanel`
   - 将阻塞原因拆解成结构化结果，而不是只显示“游客/权限”。

### 页面结构

```text
数据采集页
├─ 登录态检查卡片
│  ├─ 当前状态：Ready / Blocked
│  ├─ 问题类型
│  ├─ 检查时间
│  ├─ 实际页面
│  ├─ 授权请求数 / 401数
│  └─ 操作按钮
├─ 自动恢复记录
│  ├─ 刷新 profile
│  ├─ 切换目标页
│  ├─ 触发页面交互
│  └─ 恢复结果
└─ 采集任务区
   └─ 仅当登录态 Ready 时可执行正式采集
```

## 模块三：【后端与架构设计】

### 现状问题

当前已经有：

- `scripts/menglar-capture/login-health.mjs`
- `scripts/menglar-capture/lib/login-health.mjs`
- `POST /api/menglar/login-health`

但还缺：

- 更细粒度错误分类
- 自动恢复动作链
- 紫鸟修复入口
- 最近状态和恢复记录的结构化沉淀

### 目标错误分类

建议拆分为：

| 错误类型 | 含义 |
| --- | --- |
| `login_required` | 页面已跳登录页或明确提示登录失效 |
| `guest_page_blocked` | 页面正文显示游客态或权限不足 |
| `api_auth_missing` | 页面可开，但没有带 Authorization 的业务接口请求 |
| `api_unauthorized` | 已有授权请求，但业务接口返回 `401/403` |
| `profile_not_flushed` | 刷新前后状态变化明显，怀疑紫鸟登录态未落盘 |
| `profile_locked` | profile 副本不可用或被占用 |
| `browser_blocked` | 浏览器启动失败 |
| `unknown` | 其余未识别问题 |

### 自动恢复链路

建议顺序：

1. 基础检查
2. `refresh profile` 后重检
3. 切换目标页重检
   - `industry_general`
   - `hot_products`
4. 增加页面等待和轻量交互触发
5. 若仍失败，进入人工修复助手

### 登录修复助手

目标：

- 自动打开紫鸟到指定萌拉目标页
- 明确提示用户在紫鸟中完成登录/授权
- 后台轮询：
  - 页面 URL
  - 页面是否游客态
  - 是否出现带 Authorization 的业务请求
  - 是否有 `401/403`
- 一旦恢复为 `ready`，立即提示可以采集

### 持久化建议

建议新增缓存文件：

```text
.cache/menglar-capture/login-health-last.json
.cache/menglar-capture/login-recovery-last.json
```

后续如需前端历史展示，可再考虑落 `source_jobs` 之外的独立状态表，但本轮方案先不强制加库表。

### API 设计预留

后续建议新增：

- `POST /api/menglar/login-health/recheck`
- `POST /api/menglar/login-health/recover`
- `POST /api/menglar/login-health/open-ziniao`
- `GET /api/menglar/login-health/last`

本轮只是方案整理，不新增这些 API。

## 方案边界

本方案仅整理为后续开发目标，不在当前回合实现以下内容：

- 紫鸟环境内直接采集
- 自动登录萌拉
- 登录修复助手的真实交互闭环
- 新增恢复类 API
- 登录态恢复历史持久化表

当前阶段仍使用已有“登录态检查模块 + 数据采集页检查卡片”支持日常采集前判断。
