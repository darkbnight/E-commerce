# 商品数据整理模块字段清单

## 1. 文档目的

本清单用于明确“商品数据整理”模块在当前真实代码和数据库设计下：

- 上游 `product_content_assets` 可以提供哪些字段。
- 哪些字段需要从 `product_business_snapshots`、`source_jobs` 补充。
- 哪些发布字段不能由上游内容资产直接生成。
- 商品草稿编辑最终需要写入 `product_content_result` 的字段范围。
- 每个字段建议通过什么方式获取：`上游继承`、`规则换算`、`Ozon 接口`、`店铺配置`、`人工处理`、`AI 生成 + 人工复核`。

本清单按以下当前事实整理：

- 商品草稿编辑的新上游约定为 `product_content_assets`。
- `product_content_assets` 当前定位是商品内容资产表，保存标题、描述、属性、标签、图片 URL、图片下载结果和内容状态。
- `product_business_snapshots` 保存选品经营快照，仍用于补充品牌、类目文本、销量、销售额、尺寸、重量等字段。
- `source_jobs` 保存采集任务来源，用于批次追踪和来源回显。
- 商品草稿编辑和最终 Ozon import item 当前保存到 `product_content_result`。
- 当前本地 SQLite 中 `product_content_assets` 已建表，但暂时没有数据；热销采集脚本当前说明仍是“不主动写入该表”。所以上游真正写入 `product_content_assets` 是对接前置条件。
- 旧文档中提到的 `products_normalized`、`product_publish_drafts` 不是当前代码的主链路表；后续如恢复这些表，需要另行更新本清单。

## 2. 字段获取方式说明

| 获取方式 | 说明 |
| --- | --- |
| `上游继承` | 从 `product_content_assets`、`product_business_snapshots`、`source_jobs` 原样读取或轻度归一化 |
| `规则换算` | 根据上游字段做确定性转换，例如 `cm * 10 -> mm` |
| `Ozon 接口` | 通过 Ozon 类目、属性、字典值等接口获取 |
| `店铺配置` | 来自店铺、币种、税率、仓库等配置 |
| `人工处理` | 需要人工录入、判断、选择或复核 |
| `AI 生成 + 人工复核` | 先由 AI 生成标题、描述、属性候选等，再由人工确认 |

- `packageWidthMm`：由 `width_cm * 10` 得到
- `packageHeightMm`：由 `height_cm * 10` 得到
- `dimension_unit`：固定 `mm`
- `weight_unit`：固定 `g`

### 9.4 适合店铺配置获取的字段

- `warehouseId`

### 9.5 适合人工处理的字段

## 10. 当前最小可开工范围

### 10.3 第一阶段完成定义

第一阶段跑通以下链路即可：

```text
上游写入 product_content_assets
-> 商品数据整理工作台读取内容资产
-> 用户创建和编辑草稿
```

第一阶段不要求自动完成真实上架、库存同步、价格策略自动化或图片二次处理工作台。

## 11. 需要避免的错误映射

- 不要把 `platform_product_id` 当成 `offer_id`。
- 不要把 `sales_amount`、`average_sales_amount` 当成 `price`。
- 不要把 `category_level_1/2/3` 当成 Ozon `description_category_id`。
- 不要把竞品 `barcode` 或未知来源条码直接发布。
- 不要让上游直接写 `product_content_result`。
- 不要让 `product_content_assets` 承载价格、库存、仓库、VAT 等草稿职责。
- 不要把非 JSON 文本写入 `image_urls_json`、`attributes_json`、`downloaded_images_json`。
