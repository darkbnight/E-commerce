# 商品数据整理模块字段清单

## 1. 文档目的

本清单用于明确“商品数据整理”模块在开发时：

- 上游目前有哪些字段可以直接利用。
- 上游哪些字段暂时不能稳定复用。
- 本模块需要整理出哪些字段，才能稳定输出给下游 Ozon 上品链路。
- 每个字段建议通过什么方式获取：`AI生成`、`人工处理`、`其他方式获取`。

本清单基于以下事实整理：

- 当前项目已有上游数据源：`source_jobs`、`products_normalized`
- 当前项目已有下游执行工具：`scripts/lib/ozon-seller-client.mjs`
- 当前目标下游链路：Ozon 商品建品、价格更新、库存更新

## 2. 获取方式说明

| 获取方式 | 说明 |
| --- | --- |
| `AI生成` | 由 AI 根据上游数据、类目要求、标题/描述规则自动生成 |
| `人工处理` | 需要人工判断、录入、选择、复核或补充 |
| `其他方式获取` | 来自规则换算、接口查询、店铺配置、仓库配置、ERP、供应链资料、图片工作台等非 AI/非纯手填方式 |
| `AI生成 + 人工处理` | 先由 AI 生成初稿，再由人工复核确认 |
| `其他方式获取 + 人工处理` | 先通过接口/规则/配置拿到候选值，再由人工确认 |

## 6. 字段获取方式建议总览

### 6.1 适合 `AI生成 + 人工处理` 的字段

- `name`
- `description`
- `attributes[].values[].value`
- 后续如需：卖点摘要、营销文案、标签文案

### 6.2 适合 `人工处理` 的字段

- `price`
- `old_price`
- `premium_price`
- `min_price`
- `barcode`
- 部分高风险属性值，如材质、品牌、产地、型号

### 6.3 适合 `其他方式获取` 的字段

- `source_job_id`
- `product_normalized_id`
- `category_id`
- `attributes[].id`
- `attributes[].values[].dictionary_value_id`
- `currency_code`
- `warehouse_id`
- `dimension_unit`
- `weight_unit`

### 6.4 适合 `其他方式获取 + 人工处理` 的字段

- `offer_id`
- `vat`
- `depth`
- `width`
- `height`
- `weight`
- `images[]`
- `stock`
- `brand` 作为属性候选值

## 7. 建议的落地顺序

1. 先固定草稿表字段和接口契约。
2. 先接入上游可直接利用字段：任务来源、商品基础信息、类目文本、尺寸重量参考。
3. 再补齐下游强必需字段：`offer_id`、`name`、`category_id`、`attributes[]`、`images[]`、`price`、`currency_code`、`vat`、包装尺寸重量。
4. 最后接价格扩展、库存和导出。

## 8. 当前最小可开工范围

若只考虑“商品数据整理”模块前期开工，建议第一批先实现以下字段：

- 上游继承：
  - `source_job_id`
  - `product_normalized_id`
  - `ozon_product_id`
  - `brand`
  - `category_level_1`
  - `category_level_2`
  - `category_level_3`
  - `length_cm`
  - `width_cm`
  - `height_cm`
  - `weight_g`
- 草稿整理：
  - `offer_id`
  - `name`
  - `description`
  - `category_id`
  - `price`
  - `currency_code`
  - `vat`
  - `images[]`
  - `attributes[]`
  - `depth`
  - `width`
  - `height`
  - `weight`

这样做可以先打通“候选商品 -> 发布草稿 -> 下游导出”的最小闭环。
