# 萌拉登录态稳定性检查模块

## 模块一：【需求概述与交互】

### 需求背景

萌拉采集之前的不稳定点，不在“页面能不能打开”，而在“真实采集脚本是否复用了可用授权链路”。此前已经出现过以下问题：

1. 登录检查偶尔通过，但正式采集脚本仍然复用旧的 profile 启动方式，导致真实采集失败。
2. 刷新 profile 时直接删除整个稳定副本目录，遇到浏览器锁文件会报 `EPERM/EBUSY`。
3. 检查结果只证明“页面像是登录了”，不能证明业务接口真的可调用。

### 一句话需求描述

把萌拉登录态检查升级为“可用于正式采集”的稳定链路，并让热销商品、行业数据都能通过同一套会话能力完成真实采集。

### 解决方案大纲

1. 统一稳定会话链路：
   - 稳定副本目录只作为源，不直接拿来启动浏览器。
   - 每次检查或采集前，基于稳定副本复制一份独立运行态 profile。
   - 浏览器关闭后清理运行态 profile，避免锁冲突。
2. 强化登录态检查：
   - 不只看页面标题和游客文案。
   - 必须捕获带 `Authorization` 的业务请求。
   - 必须对目标业务接口做一次真实探测，要求返回 200。
3. 修复正式采集脚本：
   - `hot-products.mjs` 不再自己管理旧的 `launchPersistentContext(profileCopy)` 链路。
   - 改为复用 `browser-session.mjs` 的稳定启动逻辑。
4. 告警降级：
   - `Cookies` 文件复制失败如果不影响稳定副本可用性，只记为 warning，不再误报成阻塞错误。

### 用户核心体验链路

1. 先执行 `login-health` 或 `preflight`。
2. 返回 `ok=true` 且 `api.probe.ok=true` 后，再执行正式采集。
3. 热销商品与行业数据采集都复用同一套稳定会话能力。
4. 如果失败，优先根据 `errorType` 和 SOP 排查，而不是继续堆功能。

## 模块二：【前端与组件设计】

本轮不新增前端页面和组件。

已有“数据采集”页的检查能力继续保留，但这次修复重点在脚本与会话稳定性，不新增页面交互。

## 模块三：【后端与架构设计】

### 本轮核心代码调整

- `scripts/menglar-capture/lib/profile-store.mjs`
  - 不再整目录删除稳定副本。
  - 只清理受控路径。
  - 新增运行态 profile 准备与清理逻辑。
  - 当稳定副本仍可用时，`profile_locked` 仅作为 warning。

- `scripts/menglar-capture/lib/browser-session.mjs`
  - 统一浏览器状态探测。
  - 每次启动基于运行态 profile 创建持久上下文。
  - 关闭 context 时自动回收运行态 profile。

- `scripts/menglar-capture/lib/login-health.mjs`
  - 登录检查必须满足三条件：
    1. 页面不是登录失效/游客态
    2. 捕获到带授权头的业务请求
    3. 对目标业务接口真实探测返回 200

- `scripts/menglar-capture/hot-products.mjs`
  - 真实采集改为复用稳定浏览器会话模块。
  - 采集前强制执行 `runPreflight({ refresh: true })`。

### API 设计

本轮未新增或修改对外 API。

已有 `POST /api/menglar/login-health` 的能力未改动调用方式，本轮只是增强其底层判断准确性与会话稳定性。

### 数据库设计

本轮不修改数据库 schema。

自测过程中会写入业务数据，用于验证真实采集链路：
- `source_jobs`
- `product_business_snapshots`
