# 萌拉登录态稳定性检查模块

## 模块一：【需求概述与交互】

### 需求背景

萌拉采集当前最大的不稳定点不是接口字段，而是登录态复用。用户在紫鸟浏览器中看到页面已登录，不等于采集脚本能够稳定拿到可用业务接口授权。最近出现过页面可看、脚本请求 `hotPage` 返回 401、采集任务入库 0 条但状态误判的情况。

### 一句话需求描述

把萌拉登录态稳定性检查抽象成独立模块，采集前只向该模块询问“当前是否可采”，不可采时返回明确原因和处理动作。

### 解决方案大纲

1. 新增登录态健康检查模块
   - 模块位置：`scripts/menglar-capture/lib/login-health.mjs`
   - 命令入口：`scripts/menglar-capture/login-health.mjs`
   - 输出文件：`.cache/menglar-capture/login-health-last.json`

2. 检查维度
   - Chrome 是否可启动。
   - 紫鸟 profile 副本是否可用。
   - 是否读到本地登录缓存。
   - 打开目标页面后是否跳登录页或游客态。
   - 是否捕获到带 Authorization 的萌拉业务接口请求。
   - 业务接口是否返回 401/403。

3. 采集脚本复用
   - `preflight` 调用登录态模块，不重复维护一套判断。
   - 后续采集脚本只在 `ok=true` 时进入正式采集。
   - `ok=false` 时输出 `errorType` 和 `nextAction`，避免创建或误判正式采集任务。

4. 处理建议
   - `login_required`：重新登录萌拉。
   - `guest_blocked`：确认账号权限。
   - `api_auth_missing`：目标页没有触发可采业务接口。
   - `api_unauthorized`：接口返回 401/403，重新登录并关闭紫鸟窗口让登录态落盘，再刷新 profile 检查。

### 用户核心体验链路

1. 用户准备采集前先执行登录态检查。
2. 模块返回 `ok=true`，再执行采集脚本。
3. 模块返回 `ok=false`，用户按 `nextAction` 处理。
4. 处理后执行 `login-health --refresh --json` 重新确认。
5. 只有登录态稳定后才进入行业或热销商品采集。

## 模块二：【前端与组件设计】

在“数据采集”页增加登录态检查卡片：

- 状态：可采 / 不可采
- 检查时间
- 页面 URL
- 授权请求数
- 401/403 数量
- 阻塞类型
- 下一步处理建议
- 操作按钮：
  - `检查是否可采`
  - `刷新 Profile 后检查`

## 模块三：【后端与架构设计】

### 新增脚本模块

- `scripts/menglar-capture/lib/login-health.mjs`
  - 导出 `checkMenglarLoginHealth(options)`。
  - 返回标准 JSON。

- `scripts/menglar-capture/login-health.mjs`
  - 命令行入口。
  - 支持 `--target`、`--refresh`、`--headless`、`--json`。

### 输出结构

```json
{
  "ok": false,
  "status": "blocked",
  "target": "hot_products",
  "storage": {
    "runtimeStorageLoaded": true
  },
  "page": {
    "title": "登录 - 萌啦数据",
    "url": "https://ozon.menglar.com/workbench/login"
  },
  "api": {
    "requestCount": 3,
    "authorizedRequestCount": 2,
    "unauthorizedResponseCount": 2
  },
  "errorType": "api_unauthorized",
  "nextAction": "重新登录并刷新 profile"
}
```

### API 影响

新增内部 API：

- `POST /api/menglar/login-health`

用途：供“数据采集”页点击触发登录态稳定性检查。该 API 不创建采集任务，不写业务数据库，只返回检查结果并写 `.cache/menglar-capture/login-health-last.json`。

### 数据库影响

本轮不新增数据库表，仅写入 `.cache/menglar-capture/login-health-last.json`。

## 模块四：【📝 联合自动化验收用例 (TDD核心)】

- [ ] 第一阶段用例：执行 `node scripts/menglar-capture/login-health.mjs --target hot_products --json`，输出包含 `ok`、`storage`、`page`、`api`、`errorType`、`nextAction`。
- [ ] 第一阶段用例：当业务接口返回 401/403 时，结果为 `ok=false` 且 `errorType=api_unauthorized`。
- [ ] 第一阶段用例：`preflight` 复用登录态模块，输出仍兼容原有 `preflight-last.json`。
- [ ] 第二阶段用例：数据采集页点击“检查是否可采”，展示是否可采和下一步处理建议。
- [ ] 第二阶段用例：数据采集页点击“刷新 Profile 后检查”，传入 `refresh=true` 并展示新检查结果。

## 页面结构与线框方案

本轮不新增页面；检查卡片放在“数据采集”页顶部：

```text
登录态检查卡片
├─ 当前状态：可采 / 不可采
├─ 检查目标：行业数据 / 热销商品
├─ 页面状态：标题 + URL
├─ 接口授权：授权请求数 / 401 数
├─ 问题类型
└─ 下一步处理按钮：刷新 profile / 重新检查
```
