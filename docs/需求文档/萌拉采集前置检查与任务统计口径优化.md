# 萌拉采集前置检查与任务统计口径优化

## 模块一：【需求概述与交互】

### 需求背景

当前萌拉采集任务页显示的失败率偏高，主要原因不是接口采集能力本身，而是以下问题混在一起：

- 登录态、游客态、浏览器权限、profile 文件锁等环境问题被记成正式采集失败。
- 行业数据和商品热榜共用 `raw_count / normalized_count` 展示口径，导致行业采集显示 `0 / 0`，容易误判为无数据。
- 每次临时排查都需要重新理解浏览器/profile/登录态链路，消耗上下文和 token。

本次目标是建立固定、可复用的采集前置检查脚本和任务统计口径，让每次正式采集前先确认环境稳定；环境不满足时不创建正式采集任务。

### 一句话需求描述

为萌拉采集增加固定 `preflight` 检查流程，并按任务类型拆分采集结果口径，减少无效失败任务和错误判断。

### 解决方案大纲

1. 新增固定脚本目录：`scripts/menglar-capture/`
   - 所有萌拉采集、前置检查、行业下钻脚本统一放在该目录下。
   - 目录内必须提供 `README.md`，固定说明命令、使用顺序、输入输出和错误类型。
   - 后续 AI 或人工执行采集时优先查看 README，避免每次重新分析脚本链路。

2. 新增前置检查脚本：`scripts/menglar-capture/preflight.mjs`
   - 检查 Chrome 是否可启动。
   - 检查 profile 副本是否存在、必要登录态文件是否可读。
   - 打开萌拉目标页，判断是否游客态或登录失效。
   - 捕获业务接口请求头，确认是否具备调用接口条件。
   - 输出稳定 JSON 结果，供后续采集脚本复用。

3. 改造采集脚本
   - 将热销商品采集统一迁移到 `scripts/menglar-capture/hot-products.mjs`。
   - 将行业二次分析采集统一迁移到 `scripts/menglar-capture/industry-deep-dive.mjs`。
   - 删除旧入口脚本，避免后续维护两套命令。
   - 正式采集前调用或复用 preflight 结果。
   - preflight 不通过时，不创建正式 `source_jobs` 采集任务。
   - 失败原因归类为 `login_required`、`guest_blocked`、`profile_locked`、`browser_blocked`、`api_auth_missing`、`unknown`。

4. 优化任务统计口径
   - 保留 `raw_count / normalized_count` 给商品热榜任务使用。
   - 为行业任务增加通用统计字段：`request_count`、`success_count`、`record_count`。
   - 行业任务展示为“请求数 / 成功数 / 子类目数”，不再显示误导性的 `0 / 0`。

5. 优化任务页展示
   - 根据 `page_type` 决定采集结果文案。
   - `hot_products`：展示“商品：原始 / 标准化”。
   - `industry_general`：展示“行业：请求 / 成功 / 子类目”。
   - 错误信息展示归类标签，完整堆栈放入详情。

### 用户核心体验链路

1. 用户或后续 AI 先查看固定说明：`scripts/menglar-capture/README.md`
2. 执行固定前置检查命令：`node scripts/menglar-capture/preflight.mjs --target industry_general`
2. 脚本返回明确结果：
   - 环境正常：可以开始采集。
   - 登录失效：先重新登录，不创建正式采集任务。
   - 浏览器/profile 问题：先处理权限或文件锁，不创建正式采集任务。
3. 环境正常后再执行行业或商品采集脚本。
4. 任务页看到的任务记录按类型展示正确采集口径。
5. 失败记录能直接看出是登录、游客态、浏览器权限、profile 锁，还是接口授权缺失。

## 模块三：【后端与架构设计】

### 脚本目录结构

新增目录：

```text
scripts/menglar-capture/
├── README.md
├── preflight.mjs
├── hot-products.mjs
├── industry-deep-dive.mjs
└── lib/
    ├── browser-session.mjs
    ├── profile-store.mjs
    ├── preflight-checks.mjs
    └── job-store.mjs
```

目录职责：

| 文件 | 说明 |
|---|---|
| `README.md` | 固定调用说明，后续采集前先读这里 |
| `preflight.mjs` | 环境前置检查入口 |
| `hot-products.mjs` | 商品热榜采集入口 |
| `industry-deep-dive.mjs` | 行业/子类目下钻采集入口 |
| `lib/browser-session.mjs` | 浏览器启动、页面打开、请求头捕获 |
| `lib/profile-store.mjs` | profile 副本检查、刷新、文件锁判断 |
| `lib/preflight-checks.mjs` | 登录态、游客态、Authorization 捕获判断 |
| `lib/job-store.mjs` | `source_jobs` 写入、更新和统计口径封装 |

### README 要求

`scripts/menglar-capture/README.md` 必须包含：

- 目录用途：专门管理萌拉采集链路。
- 推荐执行顺序：
  1. `node scripts/menglar-capture/preflight.mjs --target industry_general`
  2. `node scripts/menglar-capture/industry-deep-dive.mjs`
  3. 或 `node scripts/menglar-capture/hot-products.mjs`
- 环境变量：
  - `ZINIAO_PROFILE_DIR`
  - `CHROME_EXECUTABLE_PATH`
  - `MENGLAR_REFRESH_PROFILE`
  - `MENGLAR_TARGET_URL`
- 输出文件：
  - `.cache/menglar-capture/preflight-last.json`
  - `data/menglar-industry/deep-dive/priority-subcategories.json`
  - `data/menglar-mvp/captured-json-full.json`
- 错误类型说明和对应处理动作。
- 安全说明：不保存 Authorization、cookie、token 到长期文件。
- 兼容说明：旧入口脚本只做转调，新开发优先使用本目录。

### 新增脚本

#### `scripts/menglar-capture/preflight.mjs`

建议命令：


### 修改采集脚本

- `scripts/menglar-capture/hot-products.mjs`
- `scripts/menglar-capture/industry-deep-dive.mjs`

处理规则：

- 默认先读取最近一次 preflight 结果，或在脚本内部执行 preflight。
- preflight 不通过时直接退出，不插入正式 `source_jobs`。
- preflight 通过后才插入正式任务。
- 行业采集完成后写入：
  - `request_count`
  - `success_count`
  - `record_count`
  - `error_type`

### 数据库 Schema 变更

表：`source_jobs`

新增字段：

| 字段 | 类型 | 说明 |
|---|---|---|
| `request_count` | INTEGER NOT NULL DEFAULT 0 | 请求次数，行业任务代表类目请求数 |
| `success_count` | INTEGER NOT NULL DEFAULT 0 | 成功响应次数 |
| `record_count` | INTEGER NOT NULL DEFAULT 0 | 业务记录数，行业任务代表子类目数，商品任务代表商品数 |
| `error_type` | TEXT | 归类后的错误类型 |

兼容策略：

- 老任务缺字段时默认 0 或 null。
- 商品热榜继续保留 `raw_count / normalized_count`。
- 新页面优先读取新字段；没有新字段时回退旧字段。

### API 修改

#### `GET /api/jobs`

返回字段新增：

| 字段 | 类型 | 说明 |
|---|---|---|
| `request_count` | number | 请求次数 |
| `success_count` | number | 成功次数 |
| `record_count` | number | 业务记录数 |
| `error_type` | string\|null | 错误分类 |

需要同步维护：`docs/API接口文档.md`

### 错误类型标准

| error_type | 含义 | 建议动作 |
|---|---|---|
| `login_required` | 登录失效 | 重新登录萌拉 |
| `guest_blocked` | 游客态访问限制 | 确认账号权限 |
| `profile_locked` | profile 文件被占用 | 关闭占用浏览器或刷新 profile 副本 |
| `browser_blocked` | 浏览器启动失败或权限不足 | 使用系统 Chrome / 检查权限 |
| `api_auth_missing` | 未捕获业务接口 Authorization | 先访问目标页并确认接口请求 |
| `db_error` | 数据库写入或读取异常 | 检查 DB 路径和写权限 |
| `unknown` | 未归类错误 | 查看详情 |

## 模块四：【📝 联合自动化验收用例 (TDD核心)】

### 第一阶段用例（Node 接口与数据库闭环）

- [ ] 编写 `scripts/menglar-capture/README.md`，断言包含固定执行顺序、环境变量、输出文件、错误类型和安全说明。
- [ ] 编写 `scripts/menglar-capture/preflight.mjs --target industry_general --json`，断言输出 JSON 包含 `ok`、`status`、`targetUrl`、`nextAction`。
- [ ] 模拟登录态缺失，执行 preflight，断言 `ok=false`、`errorType` 为 `login_required` 或 `guest_blocked`，并且不新增 `source_jobs` 正式任务。
- [ ] 模拟浏览器不可启动，执行 preflight，断言 `ok=false`、`errorType=browser_blocked`，并且不新增 `source_jobs` 正式任务。
- [ ] 执行数据库迁移后，直连查询 `source_jobs`，确认字段存在：
  - 表名：`source_jobs`
  - 字段：`request_count`、`success_count`、`record_count`、`error_type`
  - 预期：字段存在，默认值不破坏老数据读取。
- [ ] 构造一条 `industry_general` 成功任务，直连查询：
  - 表名：`source_jobs`
  - 字段：`page_type='industry_general'`
  - 字段：`request_count=3`
  - 字段：`success_count=3`
  - 字段：`record_count > 0`
  - 关联关系：任务页 `GET /api/jobs` 能返回同一条任务的上述字段。
- [ ] 构造一条 `hot_products` 成功任务，确认旧字段仍可用：
  - 表名：`source_jobs`
  - 字段：`raw_count`
  - 字段：`normalized_count`
  - 预期：任务页仍展示商品口径。
- [ ] 调用 `GET /api/jobs`，断言返回新增字段 `request_count`、`success_count`、`record_count`、`error_type`。

### 第二阶段用例（E2E 端到端浏览器闭环）

- [ ] 打开 `/tasks`，行业任务展示“请求 / 成功 / 子类目”，不再展示误导性 `0 / 0`。
- [ ] 打开 `/tasks`，商品任务仍展示“原始 / 标准化”。
- [ ] 点击失败任务详情，页面展示错误类型、完整错误和建议动作。
- [ ] 点击筛选“失败”，错误类型标签仍可见。
- [ ] 执行 preflight 成功后再执行采集脚本，任务页出现正式采集任务。
- [ ] 执行 preflight 失败场景，任务页不新增正式采集任务，仅终端输出结构化失败原因。
