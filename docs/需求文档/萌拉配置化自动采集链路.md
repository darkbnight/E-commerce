# 萌拉配置化自动采集链路

## 模块一：【需求概述与交互】

### 需求背景

当前萌拉数据采集已经有可执行脚本，但采集动作仍主要由 AI 临时理解用户意图后执行。这样可以跑通单次任务，但存在三个问题：

- 采集目标、类目参数、统计周期、数量等配置不够显性，用户很难知道本次到底按什么口径采。
- 热销商品页面存在自动请求 `catId` 为空的情况，如果不显式传入 `currentCatId`、`catLevel`、`typeId`，容易误采全站热销数据。
- 每次采集依赖 AI 重新解释任务，不利于后续批量、定时、可追溯地复用。

本需求先整理方案，后续再开发。目标是把“向 AI 下发自然语言采集指令”逐步沉淀成“用户选择或创建采集任务配置，系统按标准脚本自动执行”。

### 一句话需求描述

建立一套配置化采集任务链路，让行业数据和热销商品采集都可以通过标准任务配置自动执行，并明确采集前置检查、接口参数、入库口径和任务状态。

### 解决方案大纲

1. 保留 AI 作为调度助手，但不让 AI 重新发明采集过程
   - AI 负责确认用户意图、选择任务模板、补全类目参数。
   - 固定脚本负责前置检查、接口调用、入库和任务状态写入。
   - 后续 UI 或命令行都调用同一套采集服务，不再维护多套入口。

2. 抽象采集任务配置
   - `industry_general`：行业/类目机会数据采集，采到类目层级。
   - `hot_products`：热销商品数据采集，采到商品经营快照层级。
   - 每个任务配置必须显式记录目标页面、接口路径、类目参数、统计周期、采集数量、入库目标表。

3. 标准化热销商品接口参数
   - 统计周期固定为 28 天：`dateType=TWENTY_EIGHT_DAY`。
   - 默认采集数量固定为 50 条：`pageSize=50`。
   - 指定三级类目时必须传：
     - `catId`
     - `currentCatId`
     - `catLevel`
     - `typeId`
   - 不再依赖页面自动请求里的空 `catId`。

4. 强制采集前置检查
   - 每次正式采集前先执行 `preflight`。
   - 前置检查失败时不创建正式采集任务，避免把登录态、游客态、浏览器 profile 等问题计入采集失败率。
   - 前置检查结果写入 `.cache/menglar-capture/preflight-last.json`，后续脚本和任务页可读取。

5. 后续 UI 化
   - 在“数据采集”页提供任务模板入口。
   - 用户选择任务类型、类目、统计周期、数量后创建任务。
   - 系统展示前置检查结果、执行状态、入库统计和错误归因。

### 用户核心体验链路

1. 用户进入“数据采集”页。
2. 选择采集类型：行业数据或热销商品。
3. 选择或输入类目参数，例如 `清洁抹布` 的 `catId=17030280`、`typeId=96018`、`catLevel=3`。
4. 系统默认填入标准口径：28 天、50 条。
5. 点击开始采集后，系统先执行前置检查。
6. 前置检查通过才创建正式采集任务并入库。
7. 任务完成后，用户能在任务详情看到请求参数、入库表、记录数和失败原因。

## 模块二：【前端与组件设计】

### 新增或调整页面

- `/tasks` 或后续独立 `/capture`：数据采集页
- 任务详情弹窗或详情页：展示采集参数、前置检查、入库结果、错误信息

### 组件设计

1. `CaptureTaskTypeSelector`
   - 选择 `行业数据` 或 `热销商品`。
   - 切换后展示不同参数表单。

2. `CategoryParamForm`
   - 输入或选择 `catId`、`currentCatId`、`catLevel`、`typeId`、类目名称。
   - 对热销商品任务，三级类目必须提供 `currentCatId` 和 `catLevel`。

3. `CaptureStandardOptions`
   - 展示统计周期和数量。
   - 热销商品默认锁定为 `28 天 / 50 条`，后续可以开放高级配置。

4. `PreflightStatusPanel`
   - 展示浏览器、profile、登录态、Authorization、数据库可写状态。
   - 前置检查失败时展示下一步处理建议。

5. `CaptureTaskResultTable`
   - 展示最近任务、状态、采集结果、错误归因和操作按钮。

## 模块三：【后端与架构设计】

### 采集任务配置结构

后续可新增 `capture_task_configs` 表或先用 JSON 配置文件落地。建议字段：

| 字段 | 说明 |
| --- | --- |
| `id` | 配置主键 |
| `task_type` | `industry_general` 或 `hot_products` |
| `name` | 配置名称 |
| `target_url` | 目标页面 URL |
| `api_path` | 标准接口路径 |
| `category_name` | 类目名称 |
| `cat_id` | 类目 ID |
| `current_cat_id` | 当前类目 ID，热销商品接口需要 |
| `cat_level` | 类目层级 |
| `type_id` | 萌拉/Ozon 类型 ID |
| `date_type` | 统计周期，热销商品默认 `TWENTY_EIGHT_DAY` |
| `page_size` | 请求数量，默认 `50` |
| `enabled` | 是否启用 |
| `created_at` / `updated_at` | 创建和更新时间 |

### 标准接口

1. 行业数据
   - 接口：`/api/ozon-report-service/v1/bigDisc/industryViewV2`
   - 用途：采集类目层级的产品数、销量、销售额等机会分析数据。
   - 入库：行业类目结果或报告缓存，后续按现有行业分析链路处理。

2. 热销商品
   - 接口：`/api/ozon-report-service/v1/itemRanking/hotPage`
   - 标准请求体：

```json
{
  "catId": 17030280,
  "currentCatId": "17030280",
  "catLevel": 3,
  "typeId": 96018,
  "pageNum": 1,
  "pageSize": 50,
  "dateType": "TWENTY_EIGHT_DAY"
}
```

### 入库关系

| 表 | 作用 |
| --- | --- |
| `source_jobs` | 记录采集任务状态、请求数、成功数、入库数、错误类型 |
| `products_raw` | 保存商品原始 JSON |
| `product_business_snapshots` | 保存标准化商品经营快照 |
| `product_content_assets` | 预留商品内容资产，热销商品采集暂不主动写入 |

### API 设计预留

后续如前端化，需要新增内部 API：

- `GET /api/capture/configs`：读取采集配置列表。
- `POST /api/capture/configs`：创建采集配置。
- `POST /api/capture/preflight`：执行前置检查。
- `POST /api/capture/jobs`：按配置创建并执行采集任务。
- `GET /api/capture/jobs/:id`：查看任务详情。

本轮只整理方案并优化脚本默认接口参数，暂不新增后端 API，因此不修改 `docs/API接口文档.md`。

## 模块四：【📝 联合自动化验收用例 (TDD核心)】

- [ ] 第一阶段用例：执行热销商品采集脚本时，未传环境变量默认请求 `dateType=TWENTY_EIGHT_DAY`、`pageSize=50`。
- [ ] 第一阶段用例：传入 `MENGLAR_MAX_RECORDS=10` 时，接口请求和入库上限均为 10 条。
- [ ] 第一阶段用例：传入 `MENGLAR_HOT_CAT_ID`、`MENGLAR_HOT_TYPE_ID`、`MENGLAR_HOT_CAT_LEVEL` 后，请求体包含 `catId`、`currentCatId`、`catLevel`、`typeId`。
- [ ] 第一阶段用例：前置检查失败时不创建正式 `source_jobs` 采集任务。
- [ ] 数据库字段级验证：`source_jobs.page_type` 应为对应任务类型，`record_count` 等于本次实际入库商品数。
- [ ] 数据库字段级验证：`product_business_snapshots.job_id` 关联本次 `source_jobs.id`，`category_level_3` 与指定三级类目一致。
- [ ] 第二阶段用例：在数据采集页选择热销商品任务，默认展示 `28 天 / 50 条`。
- [ ] 第二阶段用例：前置检查失败时页面展示具体失败类型和下一步建议，不进入正式采集。

## 本轮落地范围

本轮只做两件事：

- 文档化后续“配置化自动采集链路”方案。
- 将热销商品采集脚本的标准接口口径改为 `28 天 / 50 条`。

不在本轮开发任务配置表、前端配置页面、内部采集 API 和定时任务。
