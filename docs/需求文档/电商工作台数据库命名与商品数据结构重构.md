# 电商工作台数据库命名与商品数据结构重构

## 模块一：【需求概述与交互】

### 1. 背景

当前项目数据库文件名为 `db/menglar-mvp.sqlite`，表结构以早期 MVP 采集链路为中心：

```text
source_jobs
products_raw
products_normalized
```

随着工作台从“萌拉采集验证”进入“电商数据工作台”阶段，旧命名和旧表结构已经不够准确：

- `menglar-mvp.sqlite` 绑定单一来源，且带有 MVP 临时含义。
- `products_normalized` 实际承载的是经营快照，不是稳定商品主档。
- 商品标题、描述、标签、图片等内容资产缺少独立承接表。
- 结果展示页和后续商品筛选页需要区分“经营数据”和“内容处理数据”。

### 2. 一句话需求

将数据库默认名称调整为更通用的 `ecommerce-workbench.sqlite`，并把商品数据拆成“经营快照”和“内容资产”两类，形成更清晰的采集、分析、筛选和上货准备数据底座。

### 3. 解决方案大纲

目标数据库文件：

```text
db/ecommerce-workbench.sqlite
```

目标核心表：

```text
source_jobs                    采集任务数据，暂时不调整结构
products_raw                   原始审计数据，暂时不调整结构
product_business_snapshots     商品经营快照，替代 products_normalized
product_content_assets         商品内容资产，新增
```

旧表处理：

```text
products_normalized            停止作为目标业务表使用，迁移后短期保留用于对账
```

### 4. 用户核心体验链路

1. 用户执行采集任务。
2. 系统写入 `source_jobs` 任务记录。
3. 系统保留 `products_raw` 原始载荷。
4. 系统将商品经营表现写入 `product_business_snapshots`。
5. 结果展示页按批次读取经营快照，用于初筛。
6. 用户对初筛商品进行二次处理时，系统补充或更新 `product_content_assets`。
7. 商品筛选页后续围绕内容补充、图片下载、上货状态等处理型数据展开。

## 模块二：【前端与组件设计】

### 1. 需要调整的页面

本次文档阶段不直接改页面，但后续开发时需要同步调整以下页面的数据来源和文案：

| 页面 | 调整点 |
| --- | --- |
| `/results` 结果展示页 | 从 `products_normalized` 切换为 `product_business_snapshots` |
| `/results` 商品筛选页签 | 后续关联 `product_content_assets`，展示内容、图片和处理状态 |
| 任务列表页 | 继续读取 `source_jobs`，统计字段暂不变 |

### 2. 组件影响

后续开发时建议调整：

- 结果表格字段命名：`sales` 改为 `sales_volume`，`revenue` 改为 `sales_amount`。
- 空态文案：不再出现 `products_normalized`。
- 批次商品数量统计：从 `product_business_snapshots.job_id` 统计。
- 商品筛选页后续新增内容处理列：
  - 内容状态
  - 图片状态
  - 内容更新时间
  - 操作入口：补充详情、下载图片、进入上货准备

## 模块三：【后端与架构设计】

### 1. 数据库文件命名

默认数据库文件改为：

```text
db/ecommerce-workbench.sqlite
```

历史数据库：

```text
db/menglar-mvp.sqlite
```

开发期可以直接迁移到新文件名。为了降低断链风险，建议代码提供明确报错：如果新数据库不存在，不再静默写入旧名称，而是提示执行迁移或重新采集。

### 2. 表结构设计

#### 2.1 保留表：source_jobs

暂时不调整字段结构。继续记录采集任务元信息和统计摘要。

#### 2.2 保留表：products_raw

暂时不调整字段结构。继续作为原始载荷审计表。

#### 2.3 新目标表：product_business_snapshots

替代 `products_normalized`，表示某个平台商品在某一次采集任务中的经营表现快照。

核心约束：

```text
UNIQUE(job_id, platform, platform_product_id)
```

核心关联：

```text
product_business_snapshots.job_id -> source_jobs.id
product_business_snapshots.raw_record_id -> products_raw.id
```

#### 2.4 新增表：product_content_assets

记录商品标题、描述、属性、标签、图片 URL、图片下载结果等内容资料。

核心约束：

```text
UNIQUE(platform, platform_product_id, content_hash)
```

核心关联：

```text
product_content_assets.source_job_id -> source_jobs.id
product_content_assets.source_snapshot_id -> product_business_snapshots.id
```

### 3. API 调整

后续开发需要调整以下接口内部查询，不一定改变外部返回结构：

| 接口 | 调整点 |
| --- | --- |
| `GET /api/result-jobs` | 商品数量从 `products_normalized` 改为统计 `product_business_snapshots` |
| `GET /api/products` | 商品列表从 `product_business_snapshots` 读取 |
| `GET /api/jobs` | 暂不调整 |

如果后续新增商品内容处理能力，再增加：

| 接口 | 作用 |
| --- | --- |
| `GET /api/product-content` | 查询商品内容资产 |
| `POST /api/product-content/capture` | 触发详情补采 |
| `POST /api/product-content/images/download` | 触发图片下载 |

### 4. 迁移设计

建议新增迁移脚本：

```text
scripts/migrate-ecommerce-workbench-db.mjs
```

迁移步骤：

1. 如果存在 `db/menglar-mvp.sqlite` 且不存在 `db/ecommerce-workbench.sqlite`，复制旧库为新库。
2. 在新库中创建 `product_business_snapshots`。
3. 将 `products_normalized` 历史数据迁入 `product_business_snapshots`。
4. 创建 `product_content_assets`。
5. 校验迁移前后商品记录数量。
6. 输出对账结果。

`products_normalized` 暂不立即删除，待采集脚本、后端 API、结果页和测试全部切换后再清理。

### 5. 埋点设计

本次数据库结构重构不新增前端埋点。

后续如果增加内容补采和图片下载流程，再补充：

- 商品内容补采开始/成功/失败。
- 图片下载开始/成功/失败。
- 商品进入上货准备。

## 模块四：【📝 联合自动化验收用例 (TDD核心)】

### 第一阶段用例（Node 接口与数据库闭环）

- [ ] 执行迁移脚本后，确认 `db/ecommerce-workbench.sqlite` 存在，且 `db/menglar-mvp.sqlite` 不再作为默认读写路径。
- [ ] 查库确认表 `product_business_snapshots` 已创建，字段至少包含：`job_id`、`platform`、`platform_product_id`、`sales_volume`、`sales_amount`、`captured_at`。
- [ ] 查库确认表 `product_content_assets` 已创建，字段至少包含：`platform`、`platform_product_id`、`title`、`description`、`image_urls_json`、`content_status`。
- [ ] 迁移旧数据后，查库确认 `product_business_snapshots` 记录数等于迁移前 `products_normalized` 记录数。
- [ ] 字段级验证：表 `product_business_snapshots`，字段 `job_id` 应等于原 `products_normalized.job_id`，字段 `platform_product_id` 应等于原 `products_normalized.ozon_product_id`，字段 `sales_volume` 应等于原 `products_normalized.sales`，字段 `sales_amount` 应等于原 `products_normalized.revenue`。
- [ ] 字段级验证：表 `product_business_snapshots`，唯一约束 `job_id + platform + platform_product_id` 生效；重复插入同一批次同一商品应失败或被明确忽略。
- [ ] 字段级验证：表 `product_content_assets`，插入同一 `platform + platform_product_id + content_hash` 的重复内容时，不应生成重复版本。
- [ ] 请求 `GET /api/result-jobs`，断言商品数量来自 `product_business_snapshots.job_id` 聚合。
- [ ] 请求 `GET /api/products?jobId=有效批次`，断言返回商品来自 `product_business_snapshots`，且字段映射兼容前端展示。

### 第二阶段用例（E2E 端到端浏览器闭环）

- [ ] 打开结果展示页，确认默认批次能展示来自 `product_business_snapshots` 的商品经营快照。
- [ ] 切换到空商品批次，确认空态文案不再引用 `products_normalized`。
- [ ] 使用关键词、商品类型、类目、销量、销售额筛选，确认页面结果与新表查询一致。
- [ ] 点击“将当前页加入筛选池”，确认筛选池仍能接收当前页商品。
- [ ] 切换商品筛选页签，确认二次处理页签不依赖旧 `products_normalized` 字段名。
