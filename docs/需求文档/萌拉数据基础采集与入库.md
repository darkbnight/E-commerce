# 萌拉数据基础采集与入库

## 模块一：【需求概述与交互】

### 需求背景

当前业务需要从萌拉数据网站抓取 Ozon 商品基础数据，并沉淀为本地可复用的数据资产，后续再基于这些数据做筛选、打标和分析。

现阶段的实际约束不是“有没有字段可抓”，而是“登录态依赖紫鸟浏览器时，是否仍然能通过浏览器自动化稳定完成抓取”。因此第一阶段必须先验证抓取链路和入库链路是否闭环，再进入筛选规则建设。

### 一句话需求描述

在默认已具备紫鸟浏览器登录态的前提下，通过浏览器自动化抓取萌拉数据中的商品基础数据，落库存储，并保留后续筛选扩展能力。

### 解决方案大纲

1. 使用浏览器自动化接入紫鸟浏览器中的已登录会话，验证能否稳定打开目标页面并抓取数据。
2. 以“商品基础数据”作为第一阶段目标字段，完成抓取、解析、标准化和本地入库。
3. 记录抓取任务、原始数据、标准化商品数据和字段映射，确保后续可追溯、可重放、可扩展。
4. 第一阶段不实现复杂筛选界面和规则引擎，只预留数据库结构和脚本接口。

### 用户的核心体验链路

1. 用户先在紫鸟浏览器中完成萌拉数据登录。
2. 用户启动本地抓取任务。
3. 程序连接目标浏览器上下文，进入萌拉数据目标页面。
4. 程序自动翻页或滚动抓取商品列表数据。
5. 程序将抓取结果保存到本地数据库，并输出任务结果摘要。
6. 用户后续基于数据库继续做筛选、导出或分析。

### 范围边界

- 本阶段包含：
  - 紫鸟浏览器登录态可用性验证
  - 浏览器自动化抓取
  - 商品基础字段解析
  - 本地数据库落库
  - 抓取任务日志与失败记录
- 本阶段不包含：
  - 完整筛选规则引擎
  - 人工运营后台页面
  - 多站点多账号调度
  - 自动绕过验证码、风控或二次认证

## 模块二：【前端与组件设计】

本阶段不新建业务前端页面，不强制产出《页面结构与线框方案》。

如后续需要最小可用操作界面，优先采用本地脚本入口或简单控制台输出，而不是先建设完整 Web 管理台。

若执行阶段补一个极简操作页，仅允许包含以下轻量入口：

- 抓取任务启动
- 抓取状态展示
- 最近一次任务结果摘要

当前阶段默认以前台页面自动化和本地脚本为主，不以用户可视化页面为交付中心。

## 模块三：【后端与架构设计】

### 3.1 架构原则

第一阶段采用“浏览器自动化采集 + 本地标准化 + SQLite 落库”的轻量架构，优先验证链路是否可跑通，不提前引入复杂服务拆分。

### 3.2 抓取模式设计

优先级如下：

1. 先分析萌拉数据页面的真实数据来源。
2. 若页面底层存在可复用接口，则优先直接抓接口响应。
3. 若接口受前端签名、上下文或登录态限制，则通过浏览器自动化在已登录页面中拦截响应或提取 DOM 数据。
4. 若页面需要强依赖紫鸟浏览器环境，则优先采用“连接现有浏览器实例/复用登录上下文”的方案，而不是强行用全新浏览器登录。

### 3.3 浏览器自动化方案

建议使用 Playwright。

第一阶段重点验证以下能力：

- 能否附着到紫鸟浏览器或其兼容内核实例
- 能否读取当前登录态页面
- 能否进入目标列表页并稳定分页
- 能否抓到核心字段
- 页面跳转、懒加载、翻页后数据是否完整

说明：

- 由于紫鸟浏览器属于定制环境，最大风险点不是脚本语法，而是“能否连接已登录浏览器上下文”。
- 如果无法直接附着紫鸟浏览器，则退一步验证：
  - 是否可导出/复用浏览器用户数据目录
  - 是否可通过远程调试端口启动
  - 是否必须改为半自动流程，即用户在紫鸟中打开目标页后，脚本只接管当前页采集

### 3.4 目标抓取字段

第一阶段标准字段如下：

- `ozon_product_id`：Ozon 商品 ID
- `product_type`：商品类型，枚举值建议为 `cross_border` / `local`
- `brand`：品牌，允许为空
- `category_level_1`
- `category_level_2`
- `category_level_3`
- `sales`
- `sales_growth`
- `potential_index`
- `revenue`
- `add_to_cart_rate`
- `impressions`
- `clicks`
- `view_rate`
- `ad_cost`
- `ad_cost_rate`
- `order_conversion_rate`
- `estimated_gross_margin`
- `shipping_mode`
- `delivery_time`
- `average_sales_amount`
- `length_cm`
- `width_cm`
- `height_cm`
- `weight_g`

### 3.5 数据标准化约束

- 所有百分比字段统一存为小数或数值百分比，但数据库文档必须明确口径，避免混用。
- 销量、销售额、曝光、点击等字段必须保留原始值和标准化值的转换依据。
- 类目必须拆为一二三级，无法拆分时保留原始类目字符串并标记解析状态。
- 品牌允许为空，不得因为品牌缺失导致整行数据丢弃。
- 商品唯一键以 `ozon_product_id + 抓取日期 + 数据来源页面` 为主，避免历史覆盖丢失。

### 3.6 数据库设计

建议第一阶段至少建设以下表：

1. `source_jobs`
用途：记录每次抓取任务的开始时间、结束时间、来源页面、分页方式、状态、错误摘要、抓取条数。

2. `products_raw`
用途：保存原始抓取载荷，便于回溯页面字段变化、定位解析失败和重新清洗。

3. `products_normalized`
用途：保存标准化后的商品核心字段，供后续筛选、分析和导出使用。

说明：

- 原方案中的 `source_pages` 是为了给“多个固定抓取入口”做配置复用，例如未来同时抓“热销榜”“潜力榜”“关键词榜”时，可以把页面配置单独维护。
- 但按当前阶段范围，只验证单一或少量目标页面，这层抽象偏早，先并入 `source_jobs` 更直接。

### 3.7 API 设计

第一阶段默认不对外提供正式业务 API，以本地脚本和本地数据库为主。

如果执行中需要最小接口，建议仅保留以下内部接口：

- `POST /internal/crawl/run`
  - 触发一次抓取任务
- `GET /internal/crawl/jobs`
  - 查看抓取任务列表
- `GET /internal/products`
  - 查看标准化后的商品数据

当前阶段若未实际落地上述接口，则不写入 API 接口文档。

### 3.8 埋点设计

本阶段不做前端埋点建设。

仅记录程序级运行日志：

- 抓取任务开始/结束
- 页面进入成功/失败
- 分页成功/失败
- 字段解析失败次数
- 入库成功/失败数量

## 模块四：【📝 联合自动化验收用例 (TDD核心)】

### 第一阶段用例（Node 接口与数据库闭环）

- [ ] 验证脚本可连接目标浏览器上下文，并成功打开萌拉数据目标页面；若未进入目标页，任务应标记失败，`source_jobs.job_status` 为 `failed`，且 `source_jobs.error_message` 记录具体原因。
- [ ] 验证在已登录状态下执行一次基础商品列表抓取，脚本返回成功状态，并产出非空商品列表结果。
- [ ] 验证当页面存在至少 1 条商品数据时，`products_raw` 成功写入原始记录；查库确认表 `products_raw` 的字段 `job_id`、`record_key`、`raw_payload` 均已写入，且 `job_id` 与 `source_jobs.id` 关联正确。
- [ ] 验证标准化逻辑可将商品 ID、商品类型、三级类目、销量、销售额、预估毛利率、长宽高、重量写入 `products_normalized`；查库确认表 `products_normalized` 的字段 `ozon_product_id`、`product_type`、`category_level_1`、`category_level_2`、`category_level_3`、`sales`、`revenue`、`estimated_gross_margin`、`length_cm`、`width_cm`、`height_cm`、`weight_g` 存在预期值，且 `job_id` 与 `source_jobs.id` 关联正确。
- [ ] 验证品牌为空、三级类目缺失等非关键字段异常时，脚本不会整行丢弃；查库确认 `products_normalized.ozon_product_id` 已落库，`brand` 可为空，`parse_status` 标记为 `partial` 或等效状态。
- [ ] 验证当同一商品在同一次任务中重复出现时，系统不会重复插入脏数据；查库确认表 `products_normalized` 中相同 `job_id + ozon_product_id` 仅保留一条有效记录。
- [ ] 验证当页面字段名称变化或单行解析失败时，任务不会整体崩溃；失败记录应写入 `products_raw` 或日志，`source_jobs` 记录失败数量和警告摘要。

### 第二阶段用例（E2E 端到端浏览器闭环）

- [ ] 启动脚本后，浏览器自动化能够进入萌拉数据目标页并识别列表区域。
- [ ] 执行一次分页或滚动加载后，页面中新增数据被脚本识别并采集。
- [ ] 页面出现登录失效、空数据或加载异常时，脚本能给出明确失败提示，而不是静默成功。
- [ ] 用户在紫鸟浏览器已打开目标页的情况下，脚本可完成半自动接管抓取并输出任务摘要。

### 本阶段关于筛选的约束

筛选能力延期到第二阶段建设。

第一阶段只要求：

- 数据库字段足够支撑后续筛选
- 抓取结果可被后续脚本直接读取
- 不在本阶段引入复杂筛选规则和筛选 UI
