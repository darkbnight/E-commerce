# Ozon官方物流费用规则引擎-页面结构与线框方案

## 1. 页面目标
该页面不是通用报表页，而是一个“本地物流服务比价工具页”。核心目标只有一个：
- 让你输入单个商品参数后，快速查看当前可用官方物流服务的名称、价格和配送时间

批量计算能力保留在后端 API，但第一版前端页面不承担批量任务入口。

因此页面结构应保持轻量、清晰、可解释，重点突出：
- 输入效率
- 服务列表
- 当前规则说明

## 2. 路由
- `/shipping-calculator`

## 3. 页面分区

### A. 顶部状态区
用于展示“当前系统到底在按什么规则计算”。

包含信息：
- 当前规则来源
- 最近更新时间
- 当前汇率日期
- 维护说明入口

### B. 左侧选项区
这是高频使用区，适合手工验证。

字段建议：
- 发货国家
- 仓库类型
- 销售模式
- 承运商
- 配送方法
- 商品价格
- 长
- 宽
- 高
- 重量
- 订单日期

操作按钮：
- `计算`
- `重置`

### C. 右侧服务列表区
这是最核心的输出区，必须展示“有哪些服务可用”，不能只给单个总价。

建议卡片字段：
- 服务名称
- 官方副标题
- 物流价格
- 配送时间
- 配送目标
- 是否可运输电池

### D. 底部规则说明区
用于帮助理解当前规则和后续维护。

包含：
- 当前规则文件摘要
- 当前启用方法数量
- 数据来源说明
- 更新方式说明
- 维护文档入口

## 5. 交互原则
- 单条试算优先即时反馈，不做复杂跳转。
- 服务卡片必须展示名称、价格、时间。
- 错误要明确指出是尺寸超限、重量超限、缺少规则还是缺少汇率。
- 顶部和底部都要明确告知当前规则来源，避免结果失去可解释性。

## 6. 设计取向
- 页面保持“专业工具台”风格，不做复杂运营后台布局。
- 第一版不引入批量输入区，不把页面做重。
- 后续如果批量能力确实需要人工操作，再单独扩展页面，不预埋复杂交互。
