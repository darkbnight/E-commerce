# 萌拉工作台目录重构

## 模块一：【需求概述与交互】
### 需求背景
当前仓库已经拆出了独立前端工作台和 SQLite 数据库，但 API 服务仍然留在 `tools/` 目录下，和真正的工具型脚本混在一起。这样会让项目边界继续模糊，后续扩展成一体化工作台时，前端、后端、脚本、数据库的职责不清晰。

### 一句话需求描述
将萌拉工作台的 API 服务从 `tools/` 迁出，整理为独立的 `backend/` 目录，与 `frontend/`、`db/`、`scripts/` 形成清晰分层。

### 解决方案大纲
1. 新建独立后端目录 `backend/menglar-workbench-api/`。
2. 将工作台 API 服务入口迁移到新目录。
3. 更新启动脚本、服务内部路径常量和相关文档。
4. 清理旧的 `tools/menglar-data-viewer/` 残留目录，避免双入口并存。

### 用户的核心体验链路
1. 用户执行 `npm run workbench:web:build` 构建前端。
2. 用户执行 `npm run workbench:api` 启动后端服务。
3. 服务从 `db/menglar-mvp.sqlite` 读取数据，并托管 `frontend/menglar-workbench/dist`。
4. 用户通过 `/tasks` 和 `/results` 访问工作台页面，不需要感知底层目录重构。

## 模块二：【前端与组件设计】
### 路由与前端结构
- 路由不变：
  - `/tasks`
  - `/results`
- 前端目录保持为：`frontend/menglar-workbench/`
- 本轮不新增或修改 React 组件，只调整构建产物的服务托管来源。

### 影响范围
- `vite` 构建入口继续使用 `frontend/menglar-workbench/vite.config.mjs`
- 前端请求地址继续为：
  - `GET /api/jobs`
  - `GET /api/products`

## 模块三：【后端与架构设计】
### 目录设计
- 前端目录：`frontend/menglar-workbench/`
- 后端目录：`backend/menglar-workbench-api/`
- 数据库目录：`db/`
- 脚本目录：`scripts/`
- 工具目录：`tools/` 只保留真正的工具型项目，例如 `ozon-pricing-local/`

### API 设计
- 本轮不新增、不删除、不修改 API 行为，仅迁移服务位置：
  - `GET /api/jobs`
  - `GET /api/products`

### 数据库设计
- 本轮不修改表结构
- 继续读取：
  - `source_jobs`
  - `products_raw`
  - `products_normalized`

### 服务实现
- 新服务入口：`backend/menglar-workbench-api/server.mjs`
- 数据库路径：`db/menglar-mvp.sqlite`
- 前端静态产物路径：`frontend/menglar-workbench/dist`

### 埋点设计
- 本轮不新增埋点

## 模块四：【📝 联合自动化验收用例 (TDD核心)】
### 第一阶段用例（Node 接口与数据库闭环）
- [ ] 执行 `npm run mvp:db:inspect`，确认脚本读取的是 `db/menglar-mvp.sqlite`，并成功查到 `source_jobs.id=11` 的任务记录。
- [ ] 启动 `npm run workbench:api` 后，请求 `GET /api/jobs` 返回 200，且返回体包含 `source_jobs.id`、`job_status`、`raw_count`、`normalized_count`。
- [ ] 启动 `npm run workbench:api` 后，请求 `GET /api/products` 返回 200，且返回体中的商品记录来自 `products_normalized.job_id = 最新成功任务ID`。
- [ ] 直连数据库 `db/menglar-mvp.sqlite` 校验 `products_normalized` 记录数为 10，且样本 `ozon_product_id=2107989735`、`brand='Лилия'` 存在。

### 第二阶段用例（E2E 端到端浏览器闭环）
- [ ] 执行 `npm run workbench:web:build` 后启动后端服务，访问 `/tasks` 返回 200。
- [ ] 执行 `npm run workbench:web:build` 后启动后端服务，访问 `/results` 返回 200。
- [ ] 访问 `/results` 时，前端资源从 `frontend/menglar-workbench/dist` 正常返回，不再依赖 `tools/` 目录。
