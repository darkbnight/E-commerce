# 萌拉结果工作台与商品筛选页优化

## 模块一：【需求概述与交互】
### 需求背景
当前 `/results` 同时承载了“查看原始采集结果”和“把商品推进到后续处理链路”两类目标，但 `商品筛选` 的业务定义还不够收敛，主要问题有三类：

- `结果展示` 里只有“当前页加入筛选池”，缺少“单品加入筛选池”，不适合精细挑品。
- `商品筛选` 当前被描述成偏临时池的结构，但你实际需要的是正式的选品决策工作台。
- 选品阶段的流程里，“测价”和“利润判断”本质上是同一个过程，不应拆成两个松散阶段。

如果继续沿用旧表述，后续页面、字段和接口会把“原始结果浏览”和“选品决策工作流”继续混在一起。

### 一句话需求描述
将 `/results` 明确拆成两个业务阶段：`结果展示` 只负责按批次浏览原始采集数据，并支持按页或按单品加入筛选池；`商品筛选` 负责对已入池商品执行初步筛选、测价与利润判断、供应链寻找、竞品整理，并在完成后流转给 `商品数据整理`。

### 解决方案大纲
1. 保留 `/results` 作为统一入口，但重新定义内部两个页签的职责边界。
2. `结果展示` 继续以批次为核心，只处理原始数据查看、基础筛选、分页和加入筛选池。
3. `结果展示` 同时提供两种加入入口：
   - 当前页加入筛选池
   - 单品加入筛选池
4. `商品筛选` 改为“选品决策工作台”，主视角从“批次”切换为“商品”。
5. 商品进入筛选池后，不再把批次作为主筛选条件；批次只保留为来源信息，用于回溯。
6. `商品筛选` 内部按业务阶段推进：
   - 初步筛选
   - 测价并判断是否值得做
   - 寻找供应链条
   - 整理竞品数据
   - 流转到商品数据整理
7. 顶部“入池总数”不是单独业务阶段，而是“全部”视图，用于查看所有已进入筛选池商品的整体状态分布。
8. `商品数据整理` 不参与本轮页面逻辑细化，只作为 `商品筛选` 的下游承接环节被明确引用。

### 用户核心体验链路
1. 用户进入 `/results`，默认看到 `结果展示`。
2. 用户按批次查看原始商品结果，确认这批数据中哪些商品值得进入选品流程。
3. 用户可以选择：
   - 把当前页商品加入筛选池
   - 只把单个商品加入筛选池
4. 用户切换到 `商品筛选`，此时工作对象变成“已入池商品”，而不是“当前批次商品”。
5. 用户对商品执行初步筛选，判断是否继续推进。
6. 用户对保留商品执行测价，算出成本、费用、预估售价和利润空间，并判断这个商品是否值得继续做。
7. 只有利润空间成立的商品，才继续寻找对应 1688 货源。
8. 用户把供应链和竞品信息都整理完成后，将商品流转给 `商品数据整理`。

### 范围边界
- 本轮仍保留 `/results` 单页面模式，不强制新增独立路由。
- 本轮重点是重构页面职责、交互逻辑和状态定义，不扩展 `商品数据整理` 内部设计。
- 本轮不把 `商品筛选` 直接做成上货草稿页。
- 本轮不改变采集脚本和原始采集入库链路。
- 本轮不删除来源批次信息；只是让它从前台主视角退到回溯字段。

## 模块二：【前端与组件设计】
### 路由设计
- `/results`：结果工作台，内部通过页签切换：
  - `结果展示`
  - `商品筛选`

### 页面职责
#### `结果展示`
- 只展示当前批次的原始商品经营数据。
- 支持按批次切换、基础筛选、排序、分页。
- 支持当前页加入筛选池。
- 支持单个商品加入筛选池。
- 不承担“测价”“利润判断”“找供应链”“整理竞品数据”。

#### `商品筛选`
- 只处理已进入筛选池的商品。
- 页面主视角是商品，不再是批次。
- 负责推进选品决策流程：
  - 初步筛选
  - 测价并判断是否值得做
  - 寻找供应链条
  - 整理竞品数据
  - 流转给商品数据整理
- 支持保留来源批次字段，但来源批次只用于回溯，不作为页面主结构。

### 页面区域
- 顶部页签栏：
  - 仅保留 `结果展示`、`商品筛选` 两个入口。
  - 切换到 `商品筛选` 后，页面结构和过滤条件必须明显变化。
- `结果展示` 左侧：
  - 数据批次
  - 原始数据筛选条件
- `结果展示` 右侧：
  - 统计条
  - 原始结果表格
  - `当前页加入筛选池`
  - `单品加入筛选池`
- `商品筛选` 左侧：
  - 阶段状态过滤
  - 是否已完成测价
  - 是否利润成立
  - 是否已找到供应链
  - 是否已整理竞品数据
  - 类目 / 品牌
  - 来源批次
- `商品筛选` 右侧：
  - 商品处理列表
  - 当前处理阶段
  - 测价与利润判断摘要
  - 供应链与竞品整理摘要
  - 流转到商品数据整理的动作入口

### 组件拆分建议
- `ResultModeTabs`
  - 顶部模式切换。
- `ResultBatchSelector`
  - 仅服务 `结果展示`。
- `RawProductFilterPanel`
  - 仅服务原始数据筛选。
- `RawProductTable`
  - 展示批次原始结果。
- `RawProductAddActions`
  - 承担当前页加入和单品加入筛选池操作。
- `SelectionWorkbenchFilters`
  - 服务 `商品筛选` 阶段过滤。
- `SelectionStageBoard`
  - 展示全部视图和各阶段统计。
- `SelectionProductTable`
  - 展示已入池商品及其处理字段。
- `SelectionDecisionActions`
  - 承担初筛、记录测价结果、利润判断、补充供应链、整理竞品数据、流转下游等操作。

### 状态设计
`商品筛选` 不再只用 `pending/candidate/rejected` 三个临时状态，而是改成面向业务流转的阶段状态。第一版建议至少包含：

- `pool_all`：全部视图，仅用于聚合查看所有入池商品
- `pool_pending`：已入池，待初筛
- `screening_rejected`：初筛淘汰
- `pricing_pending`：待测价与利润判断
- `pricing_rejected`：测价后利润不成立，停止推进
- `source_pending`：利润成立，待寻找供应链
- `competitor_pending`：供应链已明确，待整理竞品数据
- `prep_ready`：可流转到商品数据整理

辅助字段建议：
- `source_job_id`
- `source_platform`
- `source_platform_product_id`
- `selection_note`
- `initial_cost_price`
- `initial_delivery_cost`
- `initial_target_price`
- `initial_profit_rate`
- `pricing_decision`
- `supply_match_status`
- `supply_reference_url`
- `competitor_packet_status`

### 交互原则
- 在 `结果展示` 中，用户看的是“这一批里有什么商品”。
- 在 `商品筛选` 中，用户看的是“这些商品值不值得做，以及做到哪一步了”。
- 同一商品进入筛选池后，应避免继续被批次切换干扰主操作流。
- 若同一商品来自多个批次，页面默认按商品聚合展示，来源批次列表作为附属信息展开查看。
- 顶部“入池总数”本质上对应“全部”视图，不是额外阶段。

## 模块三：【后端与架构设计】
### API 设计调整方向
#### 继续保留
- `GET /api/result-jobs`
  - 只服务 `结果展示` 的批次选择。
- `GET /api/products`
  - 只服务 `结果展示` 的原始结果查询。

#### 后续新增建议
当 `商品筛选` 从纯前端临时态升级为真实工作流后，建议补充独立命名空间：

- `GET /api/product-selection/items`
  - 读取已进入商品筛选工作台的商品列表。
- `POST /api/product-selection/items`
  - 把原始结果中的商品加入商品筛选工作台。
- `PATCH /api/product-selection/items/:id`
  - 更新阶段状态、备注、测价结果、利润判断、供应链、竞品整理状态。
- `POST /api/product-selection/items/:id/transfer-to-prep`
  - 将商品流转到商品数据整理。

### 数据库设计建议
本轮不强制落库，但必须明确后续数据模型方向，避免继续把 `商品筛选` 建模成临时候选池。

建议后续新增表：
- `product_selection_items`

建议字段：
- `id`
- `source_job_id`
- `source_platform`
- `source_platform_product_id`
- `selection_stage`
- `selection_result`
- `selection_note`
- `initial_cost_price`
- `initial_delivery_cost`
- `initial_target_price`
- `initial_profit_rate`
- `pricing_decision`
- `supply_match_status`
- `supply_reference_url`
- `supply_vendor_name`
- `competitor_packet_status`
- `transfer_to_prep_at`
- `created_at`
- `updated_at`

字段级验证重点：
- `product_selection_items.source_job_id -> source_jobs.id`
- `product_selection_items.source_platform + source_platform_product_id` 可定位原始商品
- `product_selection_items.selection_stage` 必须落在定义枚举内
- `product_selection_items.initial_target_price` 为有效数值
- `product_selection_items.pricing_decision` 只能是“继续 / 淘汰 / 待补充”
- `product_selection_items.transfer_to_prep_at` 仅在流转时写入

### 数据关系说明
- `结果展示` 读取 `source_jobs + product_business_snapshots`
- `商品筛选` 后续读取 `product_selection_items`，并通过来源字段回溯 `product_business_snapshots`
- `商品数据整理` 后续读取已从 `product_selection_items` 流转出去的商品

### 埋点设计
本轮不新增正式埋点。

后续如果商品筛选工作流落库并进入长期使用，建议补充：
- 商品加入筛选工作台事件
- 初筛淘汰事件
- 初步价格测算完成事件
- 利润判断完成事件
- 供应链匹配完成事件
- 竞品数据整理完成事件
- 流转商品数据整理事件

## 模块四：【📝 联合自动化验收用例 (TDD核心)】
### 第一阶段用例（Node 接口与数据库闭环）
- [ ] 验证 `GET /api/result-jobs` 默认只返回可用于原始结果浏览的批次，不混入失败批次和空商品批次。
- [ ] 验证 `GET /api/products?jobId=有效批次` 返回结果来自 `product_business_snapshots`，字段可支撑结果展示原始表格。
- [ ] 验证 `结果展示` 支持单品加入筛选池，且不会误把当前页其他商品一并加入。
- [ ] 验证将商品加入商品筛选工作台时，后续接口或临时状态中至少保留 `source_job_id`、`source_platform`、`source_platform_product_id` 三个来源字段。
- [ ] 验证商品进入商品筛选后，更新阶段状态不会改写原始经营快照表 `product_business_snapshots` 的原始字段。
- [ ] 验证若后续落库 `product_selection_items`，字段 `source_job_id`、`selection_stage`、`initial_target_price`、`pricing_decision` 写入正确，且 `selection_stage` 非法值被拦截并返回 400。

### 第二阶段用例（E2E 端到端浏览器闭环）
- [ ] 打开 `/results` 后，默认进入 `结果展示`，页面能按批次查看原始商品结果。
- [ ] 在 `结果展示` 中切换批次、关键词、销量和排序后，原始结果表格按条件刷新。
- [ ] 在 `结果展示` 中点击 `当前页加入筛选池` 后，当前页商品进入 `商品筛选`。
- [ ] 在 `结果展示` 中点击单个商品的 `加入筛选池` 后，仅该商品进入 `商品筛选`。
- [ ] 切换到 `商品筛选` 后，页面主列表不再按当前批次商品整体展示，而是只展示已入池商品。
- [ ] 在 `商品筛选` 中，“全部”视图可查看所有已入池商品及其当前阶段状态。
- [ ] 在 `商品筛选` 中，商品可在“待初筛、待测价与利润判断、待找供应链、待整理竞品数据、可流转商品数据整理”等阶段之间推进。
- [ ] 在 `商品筛选` 中查看单个商品时，页面仍可查看来源批次信息，但来源批次不再作为主筛选栏的第一层控制项。
- [ ] 当商品在测价后利润不成立或被标记淘汰时，不流转到商品数据整理。
- [ ] 当商品完成竞品数据整理并标记可流转时，页面展示明确的“进入商品数据整理”入口。
