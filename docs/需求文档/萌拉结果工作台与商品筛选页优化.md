# 萌拉结果工作台与商品筛选页优化

## 模块一：【需求概述与交互】
### 需求背景
当前 `/results` 结果展示页承担了“查看采集结果”和“二次筛选商品”两个目标，但页面结构仍偏展示型：顶部大说明区占用空间，批次只能依赖默认任务，遇到 `#16` 这类行业数据或空商品批次时，用户无法快速切换到真正有商品数据的批次。同时，后续人工选品需要在采集数据中筛出有潜力上货的商品，这个流程和“结果核验”有关联，但不应该拆成两个完全独立页面，否则用户会在两个页面之间反复跳转。

### 一句话需求描述
将 `/results` 优化为单页面结果工作台：顶部用页签切换“结果展示”和“商品筛选”，结果展示作为原始数据池，商品筛选只处理从结果展示加入筛选池的商品。

### 解决方案大纲
1. 删除 `/results` 顶部大面积说明区，把当前顶部位置改为页签栏。
2. 页签栏只负责切换工作模式：
   - `结果展示`
   - `商品筛选`
3. 批次选择不放在顶部页签栏，改放在左侧筛选区顶部，作为数据范围筛选的一部分。
4. 新增批次选择能力，默认过滤空商品数量批次、失败批次和运行中批次，只展示可用于商品结果浏览的批次。
5. 结果展示模式用于查看当前批次的原始采集商品数据、统计、筛选和分页。
6. 结果展示模式提供“将当前筛选结果加入筛选池”动作，把当前页命中的商品送入商品筛选模式。
7. 商品筛选模式只展示筛选池内的商品，用于人工判断候选、排除或待判断，第一阶段只维护前端临时状态，不新增数据库表。

### 用户核心体验链路
1. 用户进入 `/results`。
2. 页面默认选中最新“成功且商品数量大于 0”的采集批次。
3. 用户可在左侧筛选区顶部切换批次，批次选择列表默认只显示批次编号、商品数量、任务类型和结束时间。
4. 用户停留在 `结果展示` 页签时，确认商品数据完整性，并通过基础筛选快速定位目标范围。
5. 用户点击“将当前结果加入筛选池”，把当前页商品加入筛选池。
6. 用户切换到 `商品筛选` 页签时，只看到筛选池中的商品。
7. 用户在商品筛选模式中将商品标记为待判断、已加入候选或已排除，形成临时潜力商品清单。

### 范围边界
- 本轮不新增 `/product-screening` 独立路由。
- 本轮不新增候选商品数据库表。
- 本轮不做 Ozon 上货草稿生成。
- 本轮不改变采集脚本入库结构。
- 本轮不处理行业数据详情页，只避免行业数据空批次干扰商品结果浏览。

## 模块二：【前端与组件设计】
### 路由设计
- `/results`：结果工作台，内部通过页签切换：
  - `结果展示`
  - `商品筛选`

### 页面职责
#### `结果展示` 页签
- 展示当前批次商品统计。
- 支持切换商品结果批次。
- 支持基础筛选、排序和分页。
- 当用户显式选择空商品批次时，展示可理解的空态提示，而不是只显示空表。

#### `商品筛选` 页签
- 只展示从结果展示页签加入筛选池的商品。
- 筛选池为空时展示引导，不默认展示全部商品。
- 复用结果展示页签的商品表格基础列。
- 增加人工筛选操作入口。
- 第一阶段支持临时状态：
  - 待判断
  - 已加入候选
  - 已排除
- 支持查看当前临时候选数量。

### 页面区域
- 顶部页签栏：
  - 只展示 `结果展示`、`商品筛选` 两个模式入口。
  - 不放批次选择器，避免顶部区域承载过多职责。
- 左侧筛选区：
  - 顶部放 `数据批次` 选择器。
  - 下方放商品筛选条件。
- 右侧内容区：
  - 上方为统计条。
  - 下方为商品表格。
  - 商品筛选页签下表格增加候选状态和操作能力。

### 组件拆分
- `ResultModeTabs`
  - 顶部页签栏。
  - 管理 `结果展示` / `商品筛选` 模式切换。
- `ResultBatchSelector`
  - 批次选择器，放在左侧筛选区顶部。
  - 默认只展示成功且商品数量大于 0 的批次。
  - 可扩展 `showUnavailable` 显示空批次或失败批次。
- `ProductMetricsBar`
  - 商品数、最高销量、最高销售额、平均毛利率、当前筛选命中数。
- `ProductFilterPanel`
  - 数据批次、关键词、商品类型、一级类目、最低销量、最低销售额、排序方式、重置筛选。
- `ProductResultTable`
  - 商品表格。
  - 支持 `readonly` 和 `screening` 两种模式。
- `ProductScreeningActions`
  - 商品筛选页签专用操作区。
  - 管理待判断、候选、排除状态。

### 表格密度优化
- 表格区域占据页面主要空间。
- 表头固定。
- 表格内部滚动，减少整页滚动。
- 行高压缩到适合密集扫描的范围。
- 数字列右对齐。
- 销量、销售额、毛利率、转化率突出可扫描性。
- 商品 ID、品牌、类目列保持稳定宽度。
- 长类目使用主类目 + 次级类目弱化显示。

### URL 状态
- `/results?mode=result&jobId=11&page=1&sort=sales_desc`
- `/results?mode=screening&jobId=11&page=1&sort=sales_desc`
- 切换页签时保留当前批次和筛选条件，避免用户重复配置。

## 模块三：【后端与架构设计】
### 新增 API
#### `GET /api/result-jobs`
用于结果工作台的数据批次选择。

请求参数：
- `includeEmpty`：是否包含商品数量为 0 的批次，默认 `false`。
- `includeFailed`：是否包含失败或运行中的批次，默认 `false`。
- `limit`：返回条数，默认 `50`，最大 `100`。

返回字段：
- `id`：任务 ID。
- `page_name`：任务名称。
- `page_type`：任务类型。
- `job_status`：任务状态。
- `raw_count`：原始记录数。
- `normalized_count`：任务记录的标准化数量。
- `product_count`：通过 `products_normalized.job_id` 实际关联出来的商品数量。
- `finished_at`：任务结束时间。

### 维护现有 API
#### `GET /api/products`
- 继续支持 `jobId`、分页、筛选和排序。
- 默认任务逻辑继续使用“最新有标准化商品数据的成功任务”。
- 建议补充返回 `actualProductCount`，便于前端区分任务记录数量和实际商品关联数量。

### 数据库设计
本轮不新增表。

继续读取：
- `source_jobs`
- `products_normalized`

关联关系：
- `source_jobs.id = products_normalized.job_id`

字段级验证重点：
- `source_jobs.id`
- `source_jobs.job_status`
- `source_jobs.page_type`
- `source_jobs.normalized_count`
- `products_normalized.job_id`
- `products_normalized.ozon_product_id`

### 后续候选商品表预留
本轮不实现，但后续如需把人工筛选结果沉淀为工作流，可新增：
- `product_candidates`

预期字段：
- `id`
- `job_id`
- `product_normalized_id`
- `ozon_product_id`
- `screening_status`
- `note`
- `created_at`
- `updated_at`

### 埋点设计
本轮不新增正式埋点。

前端保留浏览器错误输出，后端保留接口访问错误。后续如果商品筛选进入真实工作流，再补充：
- 批次切换事件。
- 模式切换事件。
- 加入候选事件。
- 排除商品事件。
- 导出候选事件。

## 模块四：【📝 联合自动化验收用例 (TDD核心)】
### 第一阶段用例（Node 接口与数据库闭环）
- [ ] 验证 `GET /api/result-jobs` 默认只返回 `job_status='success'` 且 `product_count > 0` 的批次。
- [ ] 验证 `GET /api/result-jobs?includeEmpty=true` 可返回 `product_count = 0` 的成功批次，并查库确认 `products_normalized.job_id` 无对应记录。
- [ ] 验证 `GET /api/result-jobs?includeFailed=true` 可返回失败批次，并查库确认 `source_jobs.job_status='failed'` 的记录存在。
- [ ] 验证 `GET /api/products` 默认读取最新有商品数据的成功批次，并查库确认返回的 `latestJob.id` 等于 `products_normalized.job_id` 最新可用批次。
- [ ] 验证 `GET /api/products?jobId=11` 返回 `job_id=11` 下的商品，并查库确认 `products_normalized.job_id=11`、`ozon_product_id` 与接口返回一致。
- [ ] 验证选择空商品批次时 `GET /api/products?jobId=16` 返回 200、`total=0`，且查库确认 `source_jobs.id=16` 存在但 `products_normalized.job_id=16` 记录数为 0。
- [ ] 验证 `GET /api/products?keyword=Zewa&jobId=11` 返回结果与 `products_normalized.brand='Zewa' AND job_id=11` 的查库结果一致。
- [ ] 验证 `GET /api/products?minSales=20000&jobId=11` 返回结果与 `products_normalized.sales >= 20000 AND job_id=11` 的查库结果一致。

### 第二阶段用例（E2E 端到端浏览器闭环）
- [ ] 打开 `/results` 后不再展示顶部大说明区。
- [ ] 打开 `/results` 后顶部显示 `结果展示` / `商品筛选` 页签。
- [ ] 打开 `/results` 后默认选中最新有商品数据的成功批次，页面显示商品数量和商品表格。
- [ ] 在左侧筛选区打开批次选择器，列表默认不展示空商品批次和失败批次。
- [ ] 在批次选择器中开启显示不可用批次后，可看到空商品批次，并能选择该批次。
- [ ] 选择空商品批次后，页面展示空态说明和切换到最近有商品数据批次的入口。
- [ ] 在 `结果展示` 页签设置关键词、最低销量、排序后，表格按条件刷新。
- [ ] 点击“重置筛选”后，筛选条件恢复默认，表格回到当前批次完整结果。
- [ ] 切换到 `商品筛选` 页签后，保留当前 `jobId` 和筛选条件。
- [ ] `商品筛选` 页签复用结果表格基础列，并显示筛选专用操作列或选择控件。
- [ ] 在 `商品筛选` 页签将商品加入候选、排除、恢复未处理后，页面状态和候选数量正确变化。
