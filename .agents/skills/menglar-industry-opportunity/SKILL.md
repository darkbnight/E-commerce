# 萌啦行业机会分析

用于把萌啦/Ozon 行业数据整理成可复用的类目机会分析流程。适用场景包括：

- 用户要求分析某个父级类目或子类目的三级类目机会。
- 用户提供萌啦截图、表格、接口 JSON 或要求重新采集萌啦行业数据。
- 用户需要判断哪些子类目适合早期跨境电商低成本验证。
- 用户要求把行业或类目层面的分析结果沉淀成 `docs` 下的报告。

## 工作流

### 阶段 0：采集前置检查

先判断本轮是否真的需要重新采集。

- 如果用户只要求解释公式、调整框架、改报告结构，直接使用现有文档或缓存，不启动采集。
- 如果用户要求“最新”“重新采集”“当前数据”，必须走采集权限检查。
- 如果本地已有可用缓存，先说明缓存的采集时间、父级类目和统计周期，再判断是否满足本轮分析。

采集权限检查必须确认：

- 萌啦页面可访问：`https://ozon.menglar.com/workbench/industry/general`
- 当前浏览器或本地缓存中存在有效登录态。
- 采集脚本或 Playwright 能访问目标页面。
- 网络权限、浏览器启动权限、脚本执行权限已经具备。
- 目标类目参数明确：父级类目名称、`catId`、`catLevel`、`typeId`。

如果权限或登录态不满足，先停止采集并说明缺少什么，不要继续反复请求接口。

### 阶段 1：确认分析对象

明确并记录：

- 父级类目名称
- `catId`
- `catLevel`
- `typeId`
- 统计周期
- 采集时间
- 本轮是一级分析还是三级类目下钻

不要把不同父级类目混在同一张 TopN 表里比较。

### 阶段 2：采集或读取数据

优先使用已验证可用的本地采集方式或缓存。默认行业接口为：

```text
/api/ozon-report-service/v1/bigDisc/industryViewV2
```

采集成功后只保留必要业务数据。不要把 Authorization、token、cookie、手机号、账号标识等敏感字段写入报告或长期 JSON。

### 阶段 3：字段整理和榜单合并

整理字段：

- `brandItemCount`：产品数
- `brandSales`：月销量
- `brandGmv`：月销售额
- `brandSalesRate`：销量占比
- `brandGmvRate`：销售额占比
- `brandItemCountRate`：产品数占比

如果接口同时返回销量榜、销售额榜、产品数榜，先按 `catId` 合并去重，再重新排序。

### 阶段 4：机会指数和效率计算

计算机会指数和单品效率，公式见 `references/analysis-framework.md`。

必须输出：

- 机会指数
- 单品销量
- 单品销售额

机会指数只做数据初筛，不直接等同于最终选品优先级。

### 阶段 5：早期跨境过滤

使用早期跨境选品过滤规则做逻辑筛选，规则见 `references/early-cross-border-filter.md`。

不使用复杂人工扣分制。业务风险直接归类：

- 优先验证：数据表现不错，且基本符合早期跨境过滤规则。
- 谨慎观察：数据表现好，但存在物流、合规、售后、复杂度等风险，需要继续下钻确认。
- 暂不优先：明显违反早期过滤规则，或当前团队缺少供应链、合规、售后能力。

### 阶段 6：三级类目下钻

对优先验证项继续采集到三级类目，输出新的三级类目机会排序和筛选结论。

- 三级类目
- 三级类目的产品数、月销量、月销售额
- 三级类目的机会指数和单品效率
- 三级类目的早期跨境过滤结论

如果下钻后发现类目复杂度、物流、合规或售后风险明显高于一级判断，需要调整优先级。

商品级 Top20、价格带、头部卖家集中度、毛利、物流成本、合规细节和售后细节不在本 Skill 内执行。这些内容只作为后续商品分析 Skill 的输入清单。

### 阶段 7：报告沉淀

报告只放到：

```text
docs/机会分析/萌啦行业数据/[日期]-[父级类目]-分析报告/
```

Skill 内保存方法、模板和规则；`docs` 下只保存实际分析报告。

## 关键原则

- 机会指数只做数据初筛，不等同于最终选品优先级。
- 不使用复杂人工扣分制。业务风险用逻辑筛选表达，直接归入“优先验证 / 谨慎观察 / 暂不优先”。
- 数据好但产品复杂、物流不友好、合规不确定、售后纠纷高的类目，不放入第一批验证。
- 如果用户只提供截图或表格，也按同一套字段、公式和过滤规则分析。
- 如果用户要求“最新”“重新采集”“当前数据”，必须重新采集或读取最新本地缓存，不能复用旧结论冒充最新。

## 安全要求

- 不要把 Authorization、token、cookie、手机号、账号标识等敏感字段写入报告或长期 JSON。
- 采集输出如需落盘，必须脱敏。
- 报告中只记录必要的数据口径、接口路径、请求体类目参数和分析结果。

## 输出结构

每份报告建议按 `references/report-template.md` 输出，至少包含：

- 元信息：采集页面、采集时间、请求体、接口、字段口径、机会评分公式。
- 机会指数 Top10：产品数、月销量、月销售额、机会评分、单品销量、单品销售额。
- 初步结论：优先验证、谨慎观察、暂不优先。
- 过滤理由：按物流、产品复杂度、退货纠纷、合规、供应链、差异化表达说明。
- 后续交接：把优先三级类目、风险备注、待验证问题交给商品分析 Skill。
