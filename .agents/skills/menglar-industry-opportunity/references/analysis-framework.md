# 子类目机会分析框架

本框架用于统一萌啦行业数据的子类目机会分析方法。目标不是简单找“销量最高”的类目，而是找到：

```text
需求强、市场金额足够大、供给竞争相对低，并且适合早期跨境团队验证的子类目
```

## 一、分析目标

数据层面关注：

- 月销量是否足够高
- 月销售额是否足够高
- 产品数是否相对低
- 单品效率是否健康

业务层面关注：

- 是否物流友好
- 是否产品简单
- 是否退货和纠纷风险低
- 是否合规低风险
- 国内供应链是否有优势
- 差异化是否能被页面表达

## 二、完整流程

```text
阶段 0：采集前置检查
阶段 1：确认分析对象
阶段 2：采集或读取数据
阶段 3：字段整理和榜单合并
阶段 4：机会指数和效率计算
阶段 5：早期跨境过滤
阶段 6：三级类目下钻
阶段 7：报告沉淀
```

机会指数只做数据初筛，不直接等同于最终建议优先级。

### 阶段 0：采集前置检查

先判断本轮是否需要重新采集：

- 只解释公式、调整框架、改报告结构时，直接使用现有文档或缓存。
- 用户要求“最新”“重新采集”“当前数据”时，必须重新确认采集权限。
- 本地已有缓存时，先确认采集时间、父级类目、统计周期是否满足本轮分析。

采集前必须确认：

- 萌啦页面可访问。
- 当前浏览器或缓存中存在有效登录态。
- 采集脚本或 Playwright 能访问目标页面。
- 网络权限、浏览器启动权限、脚本执行权限已经具备。
- 目标类目参数明确：父级类目名称、`catId`、`catLevel`、`typeId`。

如果权限或登录态不满足，停止采集并说明缺少什么，避免反复请求接口导致重复报错。

### 阶段 1：确认分析对象

记录当前分析对象：

```text
父级类目：
catId：
catLevel：
typeId：
统计周期：
分析层级：一级分析 / 三级类目下钻
```

不要把不同父级类目混在一起比较。

### 阶段 2：采集或读取数据

优先使用已验证可用的本地采集方式或缓存。默认接口：

```text
/api/ozon-report-service/v1/bigDisc/industryViewV2
```

采集后只保留必要业务数据，不保存 Authorization、token、cookie、手机号、账号标识等敏感字段。

### 阶段 3：字段整理和榜单合并

如果萌啦接口同时返回按销量、销售额、产品数三个榜单，先按 `catId` 合并去重，再重新计算机会指数。

### 阶段 4：机会指数和效率计算

按本框架公式计算：

- 机会指数
- 单品销量
- 单品销售额

### 阶段 5：早期跨境过滤

使用 `early-cross-border-filter.md` 做逻辑筛选，不做复杂人工扣分制。

### 阶段 6：三级类目下钻

对优先验证项继续采集到三级类目：

- 三级类目
- 三级类目的产品数、月销量、月销售额
- 三级类目的机会指数和单品效率
- 三级类目的早期跨境过滤结论

商品级 Top20、价格带、头部卖家集中度、毛利、物流成本、合规细节和售后细节不在本框架内执行，只作为后续商品分析流程的输入。

### 阶段 7：报告沉淀

报告统一存放到：

```text
docs/机会分析/萌啦行业数据/[日期]-[父级类目]-分析报告/
```

## 三、数据口径

| 字段 | 萌啦字段 | 含义 | 判断方向 |
|---|---|---|---|
| 产品数 | `brandItemCount` | 当前子类目在统计周期内的商品供给规模 | 越低越好 |
| 月销量 | `brandSales` | 当前子类目的月销量 | 越高越好 |
| 月销售额 | `brandGmv` | 当前子类目的月销售额 | 越高越好 |
| 销量占比 | `brandSalesRate` | 子类目销量在父级类目中的占比 | 越高越好 |
| 销售额占比 | `brandGmvRate` | 子类目销售额在父级类目中的占比 | 越高越好 |
| 产品数占比 | `brandItemCountRate` | 子类目产品数在父级类目中的占比 | 越低越好 |

默认接口：

```text
/api/ozon-report-service/v1/bigDisc/industryViewV2
```

采集时记录：

- 采集页面
- 采集时间
- 请求体中的类目参数
- 父级类目名称
- `catId`
- `catLevel`
- `typeId`
- 数据统计周期

## 四、机会指数公式

基础逻辑：

```text
机会 = 需求强度高 / 供给竞争低
```

推荐使用归一化加权公式：

```text
机会指数 =
0.4 × (月销量 / TopN最高月销量)
+ 0.4 × (月销售额 / TopN最高月销售额)
+ 0.2 × (1 - 产品数 / TopN最高产品数)
```

拆开看：

```text
销量得分 = 月销量 / TopN最高月销量
销售额得分 = 月销售额 / TopN最高月销售额
产品数稀缺度 = 1 - 产品数 / TopN最高产品数
```

权重解释：

| 维度 | 权重 | 原因 |
|---|---:|---|
| 月销量 | 40% | 判断需求是否真实存在 |
| 月销售额 | 40% | 判断市场金额是否足够大 |
| 产品数稀缺度 | 20% | 判断竞争密度是否相对较低 |

如果当前目标更偏低竞争，可使用保守版本：

```text
机会指数 =
0.35 × 销量得分
+ 0.35 × 销售额得分
+ 0.30 × 产品数稀缺度
```

## 五、辅助效率指标

机会指数只能做初筛，还需要看单品效率：

```text
单品销量 = 月销量 / 产品数
单品销售额 = 月销售额 / 产品数
```

判断原则：

- 机会指数高、单品销量高：需求强，且供给没有明显过度拥挤。
- 机会指数高、单品销售额高：市场金额集中，可能存在高客单机会。
- 机会指数高、单品效率低：可能是靠大量 SKU 堆量，需要谨慎。
- 销售额高但销量低：可能是高客单、重售后或重决策类目。
- 销量高但销售额低：可能是低客单、低毛利或物流成本敏感类目。

## 六、结论归类

不建议把过滤规则量化成复杂扣分表。报告中按逻辑归类：

| 结论层级 | 使用标准 |
|---|---|
| 优先验证 | 数据表现不错，且基本符合早期跨境过滤规则 |
| 谨慎观察 | 数据表现好，但存在物流、合规、售后、复杂度等风险，需要进一步下钻确认 |
| 暂不优先 | 明显违反早期过滤规则，或当前团队缺少供应链、合规、售后能力 |

## 七、分析步骤

### 第一步：确认父级类目

记录当前分析对象：

```text
父级类目：
catId：
catLevel：
typeId：
统计周期：
```

不要把不同父级类目混在一起比较。

### 第二步：整理 TopN 子类目

至少整理：

| 排名 | 子类目 | 产品数 | 月销量 | 月销售额 | 销量占比 | 销售额占比 | 产品数占比 |
|---:|---|---:|---:|---:|---:|---:|---:|

如果萌啦接口同时返回按销量、销售额、产品数三个榜单，先按 `catId` 合并去重，再重新计算机会指数。

### 第三步：计算机会指数和单品效率

按推荐公式计算机会指数，并补充：

```text
单品销量 = 月销量 / 产品数
单品销售额 = 月销售额 / 产品数
```

### 第四步：用过滤规则筛选

逐个判断 TopN 类目是否符合早期跨境选品过滤规则。重点排除：

- 物流复杂
- 产品专业性强
- 售后和纠纷风险高
- 合规或清关不确定
- 国内供应链无优势
- 差异化难表达

### 第五步：输出结论

每份分析报告必须明确：

- 机会指数 Top10 是数据初筛，不是最终建议优先级
- 优先验证哪些子类目
- 为什么优先
- 哪些子类目数据好但暂不建议做
- 哪些风险需要下钻验证
- 哪些三级类目适合交给商品分析流程继续验证

## 八、使用边界

机会指数不能直接等同于最终选品决策。进入商品开发前，还需要继续验证：

- 商品级竞争结构
- 价格带、毛利、物流体积和重量
- 合规、认证、售后和季节性风险
- 头部卖家集中度和广告成本
- 商品差异化空间
- 供应链稳定性和上新速度
